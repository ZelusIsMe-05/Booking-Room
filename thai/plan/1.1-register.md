# 1.1. Đăng ký tài khoản

## Mục tiêu

Xây dựng API đăng ký tài khoản người dùng mới bằng email, số điện thoại và mật khẩu. Sau khi đăng ký, tài khoản ở trạng thái `INACTIVE` và hệ thống gửi OTP để người dùng xác thực.

## Luồng hoạt động

1. Client gửi thông tin đăng ký đến backend.
2. Backend validate dữ liệu đầu vào.
3. Backend kiểm tra email và số điện thoại chưa tồn tại.
4. Backend lấy role `TENANT`.
5. Backend hash mật khẩu bằng bcrypt.
6. Backend tạo user với `status = INACTIVE`.
7. Backend tạo record `tenants`.
8. Backend tạo record `account_security`.
9. Backend sinh OTP 6 số, hash OTP và lưu vào `otp_verifications`.
10. Backend gửi OTP qua email.
11. Backend trả response thành công, không trả password hash và không trả OTP raw.

## API

| Thuộc tính | Giá trị |
| --- | --- |
| Method | `POST` |
| Path | `/api/auth/register` |
| Auth required | Không |
| Content-Type | `application/json` |

## Header

```http
Content-Type: application/json
Accept: application/json
```

## Request body

```json
{
  "fullName": "Nguyen Van A",
  "email": "vana@example.com",
  "phoneNumber": "0912345678",
  "password": "Password@123",
  "confirmPassword": "Password@123"
}
```

## Request schema

| Field | Type | Required | Rule |
| --- | --- | --- | --- |
| `fullName` | string | Có | Không rỗng, tối đa 255 ký tự |
| `email` | string | Có | Đúng định dạng email, unique trong `users.email` |
| `phoneNumber` | string | Có | 10 số, đúng đầu số hợp lệ, unique trong `users.phone_number` |
| `password` | string | Có | Tối thiểu 8 ký tự, có chữ hoa, chữ thường, số, ký tự đặc biệt |
| `confirmPassword` | string | Có | Phải trùng `password` |

## HTTP code

| Code | Trường hợp |
| --- | --- |
| `201` | Đăng ký thành công, OTP đã được gửi |
| `400` | Dữ liệu không hợp lệ |
| `409` | Email hoặc số điện thoại đã tồn tại |
| `500` | Lỗi server hoặc lỗi gửi OTP không xử lý được |

## Response thành công

```json
{
  "success": true,
  "message": "Đăng ký thành công. Vui lòng kiểm tra email để nhận mã OTP.",
  "data": {
    "userId": "9d971ee5-7e97-4d6c-b70b-d9a7a91b5a00",
    "email": "vana@example.com",
    "phoneNumber": "0912345678",
    "status": "INACTIVE",
    "otpExpiresInSeconds": 300
  }
}
```

## Response lỗi validation

```json
{
  "success": false,
  "message": "Dữ liệu đăng ký không hợp lệ.",
  "errors": [
    {
      "field": "password",
      "message": "Mật khẩu phải có tối thiểu 8 ký tự, chữ hoa, chữ thường, số và ký tự đặc biệt."
    }
  ]
}
```

## Response lỗi trùng email/phone

```json
{
  "success": false,
  "message": "Email hoặc số điện thoại đã được sử dụng."
}
```

## Response schema

| Field | Type | Ghi chú |
| --- | --- | --- |
| `success` | boolean | Trạng thái request |
| `message` | string | Thông báo tổng quát |
| `data.userId` | uuid | ID user mới tạo |
| `data.email` | string | Email đã đăng ký |
| `data.phoneNumber` | string | Số điện thoại đã đăng ký |
| `data.status` | string | Luôn là `INACTIVE` sau đăng ký |
| `data.otpExpiresInSeconds` | number | `300` giây |
| `errors[]` | array | Chỉ có khi validation lỗi |

## Bảng liên quan

Phần này đã được đối chiếu với các migration trong `backend/db/migrations`.

Trường hiện có:

| Field | Type/constraint | Dùng trong đăng ký |

| `full_name` | string(255), not null | Lấy từ `fullName` |
| `email` | string(255), not null, unique | Lấy từ `email` |
| `phone_number` | string(20), nullable, unique | Lấy từ `phoneNumber`; backend vẫn validate required |
| `gender` | enum `MALE/FEMALE/OTHER`, default `OTHER` | Không cần truyền khi đăng ký |
| `date_of_birth` | date, nullable | Không dùng trong đăng ký |
| `address` | string(500), nullable | Không dùng trong đăng ký |
| `avatar_url` | string(2048), nullable | Không dùng trong đăng ký |
| `status` | enum `ACTIVE/INACTIVE/BANNED`, default `ACTIVE` | Phải insert rõ `INACTIVE` |
| `username` | string(50), not null, unique | Backend tự sinh từ email/phone |
| `password` | string(255), not null | Lưu bcrypt hash |
| `role_id` | UUID, FK `roles.role_id`, not null | Role `TENANT` |

Ghi mới khi đăng ký:

- `user_id`
- `full_name`
- `email`
- `phone_number`
- `status = INACTIVE`
- `username`
- `password`
- `role_id`

Lưu ý cần chốt:

- `users.status` đang default `ACTIVE`, nhưng đăng ký OTP phải insert explicit `INACTIVE`.
- `users` hiện chưa có `created_at`/`updated_at`, nên chưa có mốc thời gian tạo đăng ký ổn định để cleanup sau 15 phút.

### `otp_verifications`

Nguồn schema: `backend/db/migrations/020_required_database_fixes.js`

Trường hiện có:

| Field | Type/constraint | Dùng trong đăng ký |
| --- | --- | --- |
| `otp_id` | UUID, primary key, default `gen_random_uuid()` | ID OTP |

Ghi mới khi đăng ký:

- `otp_id`
- `user_id`

- `phone_number`

Lưu ý cần chốt:

## Bảng/trường còn thiếu hoặc nên bổ sung

Theo schema migration hiện tại, backend có đủ bảng tối thiểu để làm đăng ký OTP nếu service xử lý chặt. Tuy nhiên, để khớp đầy đủ yêu cầu nghiệp vụ và tránh logic phụ thuộc vào suy đoán, nên bổ sung các điểm sau.

### Bắt buộc nên bổ sung: timestamp cho `users`

Lý do:

- Yêu cầu nghiệp vụ cần dọn dẹp/xóa/vô hiệu hóa đăng ký chưa xác thực sau 15 phút.
- `users` hiện không có `created_at`, nên không xác định được chính xác thời điểm tạo đăng ký.
- Dùng `otp_verifications.created_at` chỉ là phương án thay thế tạm thời; nếu user resend OTP, mốc OTP mới có thể làm lệch mốc 15 phút của bản ghi đăng ký.

Đề xuất migration:

```js
exports.up = async function (knex) {
  await knex.schema.alterTable('users', (table) => {
    table.timestamp('created_at').notNullable().defaultTo(knex.fn.now());
    table.timestamp('updated_at').notNullable().defaultTo(knex.fn.now());
  });
};

exports.down = async function (knex) {
  await knex.schema.alterTable('users', (table) => {
    table.dropColumn('updated_at');
    table.dropColumn('created_at');
  });
};
```

### Nên bổ sung: timestamp xác thực tài khoản

Lý do:

- `status = ACTIVE` cho biết tài khoản đã kích hoạt, nhưng không lưu thời điểm xác thực OTP.
- Hồ sơ người dùng có yêu cầu hiển thị trạng thái xác thực tài khoản.
- Audit/debug sẽ dễ hơn nếu có mốc xác thực trực tiếp trên user.

Đề xuất trường:

- `users.verified_at timestamp nullable`

Khi OTP đăng ký hợp lệ:

- set `status = ACTIVE`
- set `verified_at = now`
- set `updated_at = now`

### Nên bổ sung: kiểm soát OTP active

Lý do:

- Bảng `otp_verifications` cho phép nhiều OTP `REGISTRATION` chưa consumed cho cùng một user.
- Nếu không có constraint, service phải tự chọn OTP mới nhất và vô hiệu OTP cũ.

Đề xuất một trong hai cách:

1. Chỉ xử lý ở service: trước khi tạo OTP mới, set `consumed_at = now` cho OTP cũ chưa consumed cùng `user_id + purpose`.
2. Bổ sung partial unique index:

```sql
CREATE UNIQUE INDEX IF NOT EXISTS otp_one_active_registration_per_user
ON otp_verifications (user_id, purpose)
WHERE consumed_at IS NULL;
```

Nếu dùng partial unique index, resend OTP phải vô hiệu OTP cũ trước rồi mới insert OTP mới.

### Có thể bổ sung sau: tracking gửi OTP/email

Lý do:

- API đăng ký có bước gửi OTP qua email, nhưng schema hiện chưa lưu trạng thái gửi email.
- Nếu cần debug provider email hoặc retry, cần log gửi tin.

Đề xuất bảng tùy chọn:

```text
auth_delivery_logs
- delivery_id uuid primary key
- user_id uuid nullable references users(user_id)
- otp_id uuid nullable references otp_verifications(otp_id)
- channel enum EMAIL/SMS
- recipient varchar(255)
- purpose varchar(50)
- provider varchar(100) nullable
- provider_message_id varchar(255) nullable
- status enum PENDING/SENT/FAILED
- error_message text nullable
- created_at timestamp default now
- sent_at timestamp nullable
```

MVP có thể chưa cần bảng này nếu gửi OTP đồng bộ và log lỗi bằng application log.

## Điều kiện cần để thực hiện

1. Database đã chạy migration tạo `roles`, `users`, `tenants`, `account_security`, `otp_verifications`.
2. Seed đã có role `TENANT`.
3. Env có `DATABASE_URL`, `JWT_SECRET`.
4. Env có cấu hình gửi email nếu dùng OTP email thật.
5. `BCRYPT_SALT_ROUNDS >= 10`.
6. Nên bổ sung `users.created_at` để cleanup đăng ký chưa xác thực đúng 15 phút theo yêu cầu.

## Kết quả mong đợi

1. User mới được tạo ở trạng thái `INACTIVE`.
2. Password được hash bằng bcrypt.
3. Email và số điện thoại không bị trùng.
4. OTP được hash trước khi lưu.
5. OTP có hạn 5 phút.
6. Không trả password hash hoặc OTP raw trong response.

## Ghi chú triển khai

- Nên thực hiện insert user, tenant, account_security và otp trong transaction.
- Nếu gửi email OTP thất bại, cần rollback transaction hoặc đánh dấu trạng thái gửi OTP thất bại rõ ràng.
- Username có thể sinh từ phần trước `@` của email, nếu trùng thì thêm hậu tố ngắn.
