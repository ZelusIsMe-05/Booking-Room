# 1.1. Đăng ký tài khoản

## Mục tiêu

Xây dựng API đăng ký tài khoản người dùng mới bằng email, số điện thoại và mật khẩu. Sau khi đăng ký, tài khoản ở trạng thái `INACTIVE` và hệ thống gửi OTP để người dùng xác thực.

> **OTP lưu ở Redis, không dùng bảng `otp_verifications`.** OTP là dữ liệu tạm, tự hết hạn — Redis với TTL là lựa chọn phù hợp (không cần job dọn OTP, không cần unique index "1 OTP active").

## Luồng hoạt động

1. Client gửi thông tin đăng ký đến backend.
2. Backend validate dữ liệu đầu vào.
3. Backend kiểm tra email, số điện thoại và username chưa tồn tại.
4. Backend lấy role `TENANT`.
5. Backend hash mật khẩu bằng bcrypt.
6. Backend tạo user với `status = INACTIVE`.
7. Backend tạo record `tenants`.
8. Backend tạo record `account_security`.
9. Backend sinh OTP 6 số, lưu vào **Redis** key `otp:REGISTRATION:{email}` với value `{ code, attempts: 0 }`, TTL `300` giây.
10. Backend gửi OTP qua email.
11. Backend trả response thành công, không trả password hash và không trả OTP raw.

> Bước 6–8 (ghi DB) nên nằm trong transaction. Bước 9 (ghi Redis) thực hiện **sau khi commit** transaction; nếu ghi Redis/gửi email thất bại thì xử lý như mục Ghi chú triển khai.

## Request body

```json
{
  "fullName": "Nguyen Van A",
  "username": "vana",
  "email": "vana@example.com",
  "phoneNumber": "0912345678",
  "password": "Password@123",
  "confirmPassword": "Password@123"
}
```

## Request schema

| Field | Type | Required | Rule |
| --- | --- | --- | --- |
| `fullName` | string | Có | Không rỗng, tối đa 255 ký tự |
| `username` | string | Có | 3–50 ký tự, normalize lowercase, regex `^[a-z][a-z0-9_.]{2,49}$` (bắt đầu bằng chữ; chỉ chữ thường/số/`_`/`.`), unique trong `users.username` |
| `email` | string | Có | Đúng định dạng email, unique trong `users.email` |
| `phoneNumber` | string | Có | 10 số, đúng đầu số hợp lệ, unique trong `users.phone_number` |
| `password` | string | Có | Tối thiểu 8 ký tự, có chữ hoa, chữ thường, số, ký tự đặc biệt |
| `confirmPassword` | string | Có | Phải trùng `password` |

## HTTP code

| Code | Trường hợp |
| --- | --- |
| `201` | Đăng ký thành công, OTP đã được gửi |
| `400` | Dữ liệu không hợp lệ |
| `409` | Email, số điện thoại hoặc username đã tồn tại |

### OTP — lưu ở Redis (thay cho bảng `otp_verifications`)

OTP **không** ghi vào DB. Lưu trong Redis:

| Thuộc tính | Giá trị |
| --- | --- |
| Key | `otp:{purpose}:{identifier}` → đăng ký: `otp:REGISTRATION:{email}` |
| `identifier` | Email đã normalize (trim + lowercase). Đây là subject của OTP đăng ký |
| Value (logic) | `{ "code": "123456", "attempts": 0 }` — lưu plaintext |
| Cách lưu | **Redis Hash**: `HSET key code <code> attempts 0` rồi `EXPIRE key 300` |
| TTL | `300` giây (5 phút) — tự hết hạn, không cần cleanup |

Vì sao dùng Redis Hash thay vì JSON string:

- `HINCRBY key attempts 1` để tăng số lần nhập sai một cách **atomic** (xem `1.2-verify-otp.md`).
- Ghi field trên key đã tồn tại **không reset TTL** → cửa sổ 5 phút tính từ lúc tạo OTP, nhập sai không kéo dài hạn.

Ghi khi đăng ký:

- Sinh OTP 6 số, lưu plaintext vào Redis: `HSET otp:REGISTRATION:{email} code <code> attempts 0`.
- `EXPIRE otp:REGISTRATION:{email} 300` (hoặc gộp bằng pipeline/`MULTI`).
- Key trùng sẽ bị ghi đè → resend (`1.3`) chỉ cần ghi đè, không phải vô hiệu bản cũ.

Ghi chú:

- OTP lưu **plaintext** trong Redis (đã chốt). Redis là store tạm, quyền truy cập hạn chế, OTP sống 5 phút.
- Verify (`1.2`) so khớp trực tiếp `code` client gửi với `code` trong Redis. Không trả/log OTP trong response.

## Bảng/trường còn thiếu hoặc nên bổ sung

Theo schema migration hiện tại, backend có đủ bảng tối thiểu để làm đăng ký OTP nếu service xử lý chặt. Tuy nhiên, để khớp đầy đủ yêu cầu nghiệp vụ và tránh logic phụ thuộc vào suy đoán, nên bổ sung các điểm sau.

### Bắt buộc nên bổ sung: timestamp cho `users`

Lý do:

- Yêu cầu nghiệp vụ cần dọn dẹp/xóa/vô hiệu hóa đăng ký chưa xác thực sau 15 phút.
- `users` hiện không có `created_at`, nên không xác định được chính xác thời điểm tạo đăng ký.
- Vì OTP nay nằm ở Redis (TTL 5 phút) và không lưu DB, không thể dựa vào mốc OTP để tính 15 phút — **bắt buộc** cần `users.created_at` làm mốc cleanup đáng tin.

Đề xuất migration:

```js
exports.up = async function (knex) {
  await knex.schema.alterTable('users', (table) => {
    table.timestamp('created_at').notNullable().defaultTo(knex.fn.now());
    table.timestamp('updated_at').notNullable().defaultTo(knex.fn.now());
  });
};

exports.down = async function (knex) {
  await knex.schema.alterTable('users', (table) => {
    table.dropColumn('updated_at');
    table.dropColumn('created_at');
  });
};
```

### Nên bổ sung: timestamp xác thực tài khoản

Lý do:

- `status = ACTIVE` cho biết tài khoản đã kích hoạt, nhưng không lưu thời điểm xác thực OTP.
- Hồ sơ người dùng có yêu cầu hiển thị trạng thái xác thực tài khoản.
- Audit/debug sẽ dễ hơn nếu có mốc xác thực trực tiếp trên user.

Đề xuất trường:

- `users.verified_at timestamp nullable`

Khi OTP đăng ký hợp lệ:

- set `status = ACTIVE`
- set `verified_at = now`
- set `updated_at = now`

### Hạ tầng Redis cần thêm (bắt buộc cho hướng này)

- **Dependency**: `ioredis` (khuyến nghị) — `npm install ioredis`.
- **Client dùng chung**: `backend/config/redis.js` khởi tạo một client từ `REDIS_URL` (singleton, dùng lại cho mọi flow OTP và rate-limit).
- **Wrapper thao tác OTP**: `backend/redis/otpStore.js` gói các hàm:
  - `setOtp({ purpose, identifier, code })` → `HSET` + `EXPIRE` (TTL = `OTP_TTL_SECONDS`).
  - `verifyOtp({ purpose, identifier, code })` → `HGETALL` so khớp `code`; trả `{ status, attempts }`.
  - `incrAttempts({ purpose, identifier })` → `HINCRBY ... attempts 1`, trả về số attempts mới.
  - `deleteOtp({ purpose, identifier })` → `DEL` (dùng khi verify đúng hoặc vượt số lần).
  - `setResendCooldown` / `isOnCooldown` cho rate-limit resend (xem `1.3`).
- **Env mới**:
  - `REDIS_URL` (default `redis://127.0.0.1:6379`).
  - `OTP_TTL_SECONDS` (default `300`).
  - `OTP_MAX_ATTEMPTS` (default `3`).
  - `OTP_RESEND_COOLDOWN_SECONDS` (default `60`).
- **Lưu ý**: vì OTP rời khỏi DB, bảng `otp_verifications` không còn được dùng trong các flow `1.1`/`1.2`/`1.3`. Có thể giữ migration cũ (không drop) nhưng service không đụng tới; hoặc dọn ở phase sau. Job cleanup `1.7` không cần xóa OTP nữa (Redis tự hết hạn).

## Điều kiện cần để thực hiện

1. Database đã chạy migration tạo `roles`, `users`, `tenants`, `account_security` (OTP không dùng DB nữa).
2. Seed đã có role `TENANT`.
3. Env có `DATABASE_URL`, `JWT_ACCESS_SECRET`, `JWT_REFRESH_SECRET`.
4. **Redis đang chạy và `REDIS_URL` đã cấu hình** (kèm `OTP_TTL_SECONDS`, `OTP_MAX_ATTEMPTS`, `OTP_RESEND_COOLDOWN_SECONDS` nếu muốn override default).
5. Env có cấu hình gửi email nếu dùng OTP email thật.
6. `BCRYPT_SALT_ROUNDS >= 10`.
7. Nên bổ sung `users.created_at` để cleanup đăng ký chưa xác thực đúng 15 phút theo yêu cầu.

## Kết quả mong đợi

1. User mới được tạo ở trạng thái `INACTIVE`.
2. Password được hash bằng bcrypt.
3. Email và số điện thoại không bị trùng.
4. OTP được lưu trong Redis key `otp:REGISTRATION:{email}` (plaintext) với `attempts = 0`.
5. OTP có hạn 5 phút (TTL Redis tự hết hạn).
6. Không trả password hash hoặc OTP raw trong response.

## Ghi chú triển khai

- Insert user, tenant, account_security trong **một transaction DB**. OTP ghi vào Redis **sau khi commit** (Redis không nằm trong transaction DB được).
- Thứ tự đề xuất: commit DB → `setOtp` vào Redis → gửi email. Nếu `setOtp` hoặc gửi email thất bại: user vẫn tồn tại ở `INACTIVE`, client có thể gọi `resend-otp` (`1.3`) để thử lại; trả lỗi `500` rõ ràng. Không cần rollback user (tránh mất dữ liệu đăng ký vì sự cố Redis/email tạm thời).
- `identifier` trong key phải normalize nhất quán (trim + lowercase email) ở cả 3 flow để verify/resend tra đúng key.
- `phoneNumber` vẫn được thu thập và lưu (unique) như thông tin liên hệ, nhưng **không xác thực qua SMS** — chỉ email được xác thực OTP.
- `username` do user tự nhập (normalize lowercase), không suy đoán từ email nữa → tránh đụng độ giữa các email khác domain nhưng trùng phần trước `@`.
- Nên kiểm tra trùng `email` / `phone_number` / `username` và trả `409` rõ ràng. Có thể dựa vào unique constraint của DB (bắt lỗi unique violation) thay vì query trước, nhưng query trước cho message thân thiện hơn.
