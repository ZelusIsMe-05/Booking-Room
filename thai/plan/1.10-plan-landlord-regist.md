# 1.10. Tách đăng ký Tenant / Landlord + duyệt Landlord

## Mục tiêu

Cho phép người dùng **chọn vai trò ngay khi đăng ký**:

- `Tenant` (người thuê): đăng ký → xác thực OTP → dùng được ngay.
- `Landlord` (chủ nhà): đăng ký kèm **ảnh CCCD mặt trước/sau** → xác thực OTP → **chờ Admin duyệt** mới được dùng quyền chủ nhà (đăng phòng…).

Thay cho luồng cũ (mọi user mặc định là Tenant, muốn lên Landlord phải xin Admin đổi role).

## Hiện trạng code (đối chiếu)

Phần lớn hạ tầng đã sẵn sàng, chỉ cần tách endpoint + bổ sung duyệt:

- `models/User.js`: `registerSchema` **đã có** field `role: enum(['TENANT','LANDLORD']).default('TENANT')`.
- `services/auth/auth.service.js#register`: **đã nhận** `role` và truyền xuống repo.
- `repositories/auth/auth.repository.js#createUserWithRole`: **đã branch** theo `roleName` — nếu `LANDLORD` thì insert `landlords` với `id_card_front_url=''`, `id_card_back_url=''` (placeholder rỗng).
- `db/migrations/005_create_landlords.js`: `landlords(landlord_id PK→users, id_card_front_url NOT NULL, id_card_back_url NOT NULL)` — **chưa có cột trạng thái duyệt**.
- Upload: `config/multer.js` (memoryStorage, 5MB, chỉ image) + `middlewares/uploadMiddleware.js` (`upload.array('images',10)`). File được ghi ra đĩa `uploads/...` và trả path `/uploads/...` (xem `services/host/roomService.js`).
- `middlewares/validateMiddleware.js`: `validate` **ghi đè** `req.body` bằng kết quả Zod → multer phải chạy **trước** validate.
- Trạng thái user chỉ có `ACTIVE/INACTIVE/BANNED` (migration 004) — **không có** `PENDING`. Admin `services/admin/userService.js` có lock/unlock/đổi role/reset password, **chưa có** duyệt Landlord.

> Kết luận: yêu cầu khả thi với thay đổi nhỏ. Việc lớn nhất là (1) nhận + lưu ảnh CCCD thật thay placeholder rỗng, và (2) thêm trạng thái duyệt + endpoint Admin duyệt + chốt chặn quyền Landlord.

## Đề xuất giải pháp

### Quyết định thiết kế: biểu diễn "Landlord chờ duyệt"

**Chọn: thêm cột `approval_status` vào bảng `landlords`** (`PENDING | APPROVED | REJECTED`, default `PENDING`), kèm `rejection_reason`, `reviewed_at`, `reviewed_by`.

Lý do (so với phương án thêm status `PENDING_APPROVAL` vào `users`):

- `users.status` đang được dùng xuyên suốt login/refresh/verify-otp; thêm enum mới sẽ phải sửa nhiều nhánh và dễ gãy. Trạng thái duyệt là **đặc thù của Landlord** → đặt ở bảng `landlords` đúng phạm vi hơn.
- Landlord vẫn đăng nhập được sau khi verify OTP (`status=ACTIVE`), chỉ bị chặn ở **các thao tác chủ nhà** cho tới khi `approval_status=APPROVED`. Tenant không bị ảnh hưởng.

### Luồng tổng quát

**Giữ duy nhất 1 endpoint `POST /api/auth/register`.** Field `role` trong body quyết định nhánh xử lý (validate + service). Không tách `register-tenant`/`register-landlord`.

**Đăng ký Tenant** (`role=TENANT`) — giữ nguyên logic 1.1:
1. `POST /api/auth/register` (JSON, `role=TENANT`) → tạo user `INACTIVE` + `tenants` + `account_security` → gửi OTP.
2. Verify OTP → `ACTIVE` → dùng ngay.

**Đăng ký Landlord** (`role=LANDLORD`):
1. `POST /api/auth/register` (**multipart/form-data**, `role=LANDLORD`): các field như Tenant + 2 file `id_card_front`, `id_card_back`.
2. multer nhận 2 ảnh (memory) → service lưu ra đĩa → tạo user `INACTIVE` + `landlords(approval_status=PENDING, id_card_*_url=key thật)` + `account_security` → gửi OTP.
3. Verify OTP → `users.status=ACTIVE`, `landlords.approval_status` vẫn `PENDING`.
4. Admin xem hồ sơ (ảnh CCCD) → `APPROVED` hoặc `REJECTED(reason)`.
5. Trước khi `APPROVED`, mọi thao tác chủ nhà bị chặn (403).

> Vì 1 endpoint nhận cả JSON (tenant) lẫn multipart (landlord): luôn chạy `uploadIdCards` (multer `.fields()` **tự bỏ qua** khi request không phải multipart, nên tenant gửi JSON vẫn đi qua bình thường). Sau đó middleware validate **chọn schema theo `req.body.role`**.

## Thay đổi chi tiết

### 1. Routes — `routes/auth/auth.route.js`

Giữ nguyên 1 endpoint `/register`. Thứ tự middleware: multer (parse multipart, bỏ qua nếu JSON) → validate chọn schema theo role → controller.

```js
router.post(
  '/register',
  uploadIdCards,            // multer.fields; no-op khi request là JSON (tenant)
  validateRegisterByRole,   // chọn schema theo req.body.role (mục 3)
  authController.register,
);
```

### 2. Upload middleware — `middlewares/uploadMiddleware.js`

```js
const uploadIdCards = upload.fields([
  { name: 'id_card_front', maxCount: 1 },
  { name: 'id_card_back', maxCount: 1 },
]);
```

- File text khác sẽ nằm ở `req.body` (multer điền), 2 ảnh ở `req.files.id_card_front[0]` / `req.files.id_card_back[0]`.
- Lỗi multer (quá 5MB / sai mime) cần được map sang `AppError 400` (kiểm tra error handler hiện tại có bắt `MulterError` chưa; nếu chưa, thêm nhánh xử lý).

### 3. Schemas + middleware validate động — `models/User.js`

Giữ `role` trong model như yêu cầu. Tách `registerSchema` thành 2 schema body (đều **giữ** field `role` để controller dùng lại, nhưng cố định bằng literal cho rõ ràng):

- `registerTenantSchema`: các field chung + `role: z.literal('TENANT')`.
- `registerLandlordSchema`: các field chung + `role: z.literal('LANDLORD')`. Ảnh CCCD **không** validate ở Zod (là file, nằm ở `req.files`) → kiểm tra ở controller (mục 4).

> Hiện body 2 loại gần như giống hệt; tách 2 schema để sau này landlord thêm field riêng (vd số CCCD, ngày cấp) thì chỉ sửa `registerLandlordSchema`, không đụng tenant.

**Middleware `validateRegisterByRole`** (đặt ở `middlewares/` hoặc cạnh `validate`): đọc `req.body.role` để chọn schema, tái dùng cùng cơ chế ép kiểu + map lỗi của `validate` hiện tại.

```js
function validateRegisterByRole(req, res, next) {
  const role = String(req.body.role || 'TENANT').toUpperCase();
  const schema = role === 'LANDLORD' ? registerLandlordSchema : registerTenantSchema;
  // dùng lại logic của validate(): parse → ghi đè req.body → map ZodError sang AppError 400
  return validate({ body: schema })(req, res, next);
}
```

- `role` thiếu/sai → mặc định nhánh TENANT (giữ tương thích client cũ gọi `/register` không kèm role). Nếu muốn chặt hơn: validate `role` enum trước, sai thì 400.
- Cập nhật `toRegisterResponse` để trả thêm `role`, và `approvalStatus` cho Landlord.

### 4. Controller — `controllers/auth/auth.controller.js`

Một handler `register` rẽ nhánh theo `role` (đã được validate):

```js
async function register(req, res, next) {
  const { role } = req.body;
  if (role === 'LANDLORD') {
    const front = req.files?.id_card_front?.[0];
    const back  = req.files?.id_card_back?.[0];
    if (!front || !back) {
      throw new AppError('ID_CARD_REQUIRED', 'Vui lòng tải lên ảnh mặt trước và mặt sau CCCD.', 400);
    }
    const result = await authService.registerLandlord({ ...req.body, idCardFront: front, idCardBack: back });
    return sendSuccess(res, { status: 201, message: 'Đăng ký chủ nhà thành công. Vui lòng xác thực OTP; hồ sơ sẽ được Admin duyệt.', data: { ...result.user, otpExpiresInSeconds: result.otpExpiresInSeconds } });
  }
  const result = await authService.register({ ...req.body }); // TENANT (giữ nguyên)
  return sendSuccess(res, { status: 201, message: 'Đăng ký thành công. Vui lòng kiểm tra email để nhận mã OTP.', data: { ...result.user, otpExpiresInSeconds: result.otpExpiresInSeconds } });
}
```

- Việc kiểm tra **bắt buộc 2 file** cho landlord nằm ở đây (Zod không validate file). Đây là điểm khác biệt thực sự giữa 2 nhánh ngoài body.

### 5. Service — `services/auth/auth.service.js`

- Tách `register` thành 2 (hoặc dùng chung lõi):
  - `register`/`registerTenant`: như hiện tại, `role='TENANT'`.
  - `registerLandlord({ ..., idCardFront, idCardBack })`:
    1. Check trùng email/phone/username (như cũ).
    2. Lấy role `LANDLORD`.
    3. Hash password.
    4. Tạo trước `landlordId` (= `user_id`). Vì `users.user_id` mặc định sinh ở DB (`gen_random_uuid()`), cần **sinh UUID ở app** (`crypto.randomUUID()`) và truyền vào insert để biết tên thư mục ảnh **trước** khi ghi file.

> **Vì sao sinh `user_id` ở app thay vì để DB sinh:** tên thư mục ảnh `uploads/landlords/{user_id}/` cần biết `user_id` *trước* khi ghi file, mà ta muốn ghi file **trước** khi mở transaction DB. So sánh 2 hướng:
>
> | | Sinh `user_id` ở app (ghi file trước TX) — **chọn** | Ghi file trong TX (DB sinh id) |
> |---|---|---|
> | Ưu | Transaction DB ngắn, không giữ kết nối khi đang I/O đĩa/mạng; fail file thì chưa đụng DB | Không đổi cách sinh id; rollback DB tự động |
> | Nhược | Truyền id tường minh; rác file nếu TX fail → cần `idCardStorage.remove()` | Giữ transaction mở suốt lúc ghi đĩa — **khi lên S3 thì gọi mạng trong transaction là anti-pattern** |
>
> Truyền `user_id` tường minh chỉ override default cho lần insert này, **không** ảnh hưởng các insert khác (tenant/OAuth vẫn để DB sinh). Cần kiểm tra không có trigger/logic nào giả định `users` luôn insert *không kèm* `user_id`; `crypto.randomUUID()` là UUID v4 hợp lệ khớp kiểu cột `uuid`.
    5. **Lưu 2 ảnh** qua `idCardStorage.save(...)` (mục 7) → nhận `frontUrl`, `backUrl`.
    6. `createUserWithRole({ ..., userId: landlordId, roleName:'LANDLORD', idCardFrontUrl: frontUrl, idCardBackUrl: backUrl })`.
    7. Sinh + lưu OTP Redis + gửi email (như Tenant).
- Lưu ý thứ tự: ghi file **trước** transaction DB; nếu transaction fail thì xóa 2 file vừa ghi (`idCardStorage.remove(landlordId)`) để tránh rác. Không để placeholder rỗng như hiện tại.

### 6. Repository — `repositories/auth/auth.repository.js`

- `createUserWithRole`: nhận thêm `userId` (UUID sinh từ app, xem mục 5), `idCardFrontUrl`, `idCardBackUrl`. Insert `users` truyền explicit `user_id: userId`. Nhánh `LANDLORD` insert:

```js
await trx('landlords').insert({
  landlord_id: user.user_id,
  id_card_front_url: idCardFrontUrl,
  id_card_back_url: idCardBackUrl,
  approval_status: 'PENDING',
});
```

- Thêm hàm cho Admin: `findPendingLandlords()`, `findLandlordById(id)`, `updateLandlordApproval(id, { status, reason, reviewedBy })`.
- Thêm `getLandlordApprovalStatus(userId)` phục vụ guard.

### 7. Lưu ảnh CCCD — local hiện tại, sẵn sàng S3 sau

**Quy ước lưu trữ (chốt cho giai đoạn local):**

- Thư mục: `backend/uploads/landlords/{landlordId}/` (`landlordId` = `user_id`).
- Tên file cố định: `1.jpg` = **ảnh mặt trước**, `2.jpg` = **ảnh mặt sau**.
- Tên cố định `1`/`2` → đăng ký lại / cập nhật sẽ **ghi đè** đúng file, không tích rác theo timestamp.
- Lưu DB (`id_card_front_url` / `id_card_back_url`): nên lưu **key tương đối** `landlords/{landlordId}/1.jpg` (không lưu full URL) để khi chuyển S3 chỉ đổi base URL, không phải migrate dữ liệu. Tầng đọc sẽ ghép `STORAGE_PUBLIC_BASE` + key khi trả ra API.
  - (Nếu muốn đơn giản giai đoạn này: lưu luôn path `/uploads/landlords/{id}/1.jpg`. Đánh đổi: khi lên S3 phải sửa dữ liệu cũ. Khuyến nghị lưu key tương đối.)

> Ghi chú phần mở rộng: ảnh được lưu `.jpg` cố định bất kể định dạng gốc. Để chính xác, hoặc (a) ép convert sang jpeg khi lưu (cần `sharp`), hoặc (b) giữ đuôi gốc và lưu `1.<ext>` rồi ghi `ext` vào DB. Giai đoạn local có thể chấp nhận đặt tên `.jpg` theo yêu cầu; ghi rõ giả định này.

**Tầng trừu tượng để sau này thay S3 không sửa nghiệp vụ:**

- Tạo `services/storage/idCardStorage.js` (hoặc `utils/storage/`) expose interface ổn định:
  - `save({ landlordId, frontFile, backFile }) -> { frontKey, backKey }`
  - `remove(landlordId)` — xóa thư mục/đối tượng (dùng khi rollback hoặc landlord bị xóa)
  - `getReadableUrl(key)` / `getStream(key)` — phục vụ Admin xem ảnh
- Hiện tại implement bằng `fs` (ghi `backend/uploads/landlords/...`). Khi tích hợp AWS S3: thêm implementation `s3` (dùng `@aws-sdk/client-s3`, `PutObjectCommand`, presigned URL cho đọc) và chọn qua biến môi trường `STORAGE_DRIVER=local|s3`. **Service/controller không đổi** vì chỉ phụ thuộc interface.
- Env dự phòng cho S3 (chưa cần set bây giờ): `STORAGE_DRIVER`, `S3_BUCKET`, `S3_REGION`, `AWS_ACCESS_KEY_ID`, `AWS_SECRET_ACCESS_KEY`, `STORAGE_PUBLIC_BASE`.

**Bảo mật**: ảnh CCCD là dữ liệu nhạy cảm. `backend/uploads/` đang được serve tĩnh public (`app.js`). Khuyến nghị: **KHÔNG** serve `uploads/landlords` qua static; chỉ cho Admin xem qua endpoint `GET /api/admin/landlords/:id/id-card/:side` có `authorizeRoles('ADMIN')` (stream file qua `idCardStorage.getStream`). Khi lên S3 thì dùng **presigned URL** ngắn hạn thay vì public-read. Nếu giai đoạn local tạm chấp nhận để public thì ghi rõ đây là nợ kỹ thuật phải xử lý trước production.

### 8. Migration mới — bổ sung cột duyệt cho `landlords`

```js
exports.up = async (knex) => {
  await knex.schema.alterTable('landlords', (t) => {
    t.enu('approval_status', ['PENDING', 'APPROVED', 'REJECTED'],
      { useNative: true, enumName: 'landlord_approval_status' })
      .notNullable().defaultTo('PENDING');
    t.string('rejection_reason', 500).nullable();
    t.timestamp('reviewed_at').nullable();
    t.uuid('reviewed_by').nullable().references('user_id').inTable('users');
  });
};
```

- Dữ liệu landlord cũ (nếu có): cân nhắc backfill `APPROVED` để không khóa nhầm tài khoản đang hoạt động.

### 9. Chốt chặn quyền Landlord (gating)

- Thêm middleware `requireApprovedLandlord` (sau `requireAuth, authorize('LANDLORD')`): query `landlords.approval_status`, nếu `≠ APPROVED` → `403 LANDLORD_NOT_APPROVED` (kèm trạng thái PENDING/REJECTED + reason).
- Áp cho các route chủ nhà tạo/sửa tài nguyên (vd `routes/host/roomRoutes.js` POST/PATCH/DELETE). GET danh sách của chính mình có thể cho phép để họ thấy trạng thái.
- `GET /api/auth/me` nên trả thêm `approvalStatus` cho Landlord để FE hiển thị "đang chờ duyệt".

### 10. Admin duyệt Landlord — `routes/admin/userRoutes.js` + `services/admin/userService.js`

- `GET /api/admin/landlords?status=PENDING` — danh sách hồ sơ chờ duyệt.
- `GET /api/admin/landlords/:id` — chi tiết hồ sơ (trả ref ảnh, không phải full path nếu lưu key tương đối).
- `GET /api/admin/landlords/:id/id-card/:side` (`side=front|back`) — stream ảnh CCCD qua `idCardStorage.getStream` (chỉ Admin), thay cho serve tĩnh public.
- `PATCH /api/admin/landlords/:id/approve` — set `APPROVED`, `reviewed_at`, `reviewed_by`; ghi `systemLog`; (tùy chọn) gửi email/thông báo cho landlord.
- `PATCH /api/admin/landlords/:id/reject` — body `{ reason }`, set `REJECTED` + `rejection_reason`; ghi log; thông báo.
- Dùng `authorizeRoles('ADMIN')` (đã có ở `userRoutes`).

## Bảng liên quan

| Bảng | Thay đổi |
| --- | --- |
| `users` | Không đổi (Landlord vẫn `INACTIVE→ACTIVE` theo OTP) |
| `landlords` | **Thêm** `approval_status`, `rejection_reason`, `reviewed_at`, `reviewed_by`; `id_card_*_url` lưu key tương đối `landlords/{id}/1.jpg`(front) & `2.jpg`(back) thay vì `''` |
| `tenants` | Không đổi |
| `system_logs` | Ghi log hành động Admin approve/reject |

## HTTP code

| Endpoint | Code | Trường hợp |
| --- | --- | --- |
| `register-tenant` / `register-landlord` | 201 | Thành công, OTP đã gửi |
| | 400 | Validate lỗi / thiếu ảnh CCCD / file quá lớn / sai định dạng |
| | 409 | Trùng email/phone/username |
| host actions | 403 | `LANDLORD_NOT_APPROVED` khi chưa duyệt |
| admin approve/reject | 200 | Duyệt/từ chối thành công |
| | 404 | Không tìm thấy hồ sơ landlord |

## Tương thích ngược / rủi ro

- Giữ nguyên `/api/auth/register` → client cũ gọi không kèm `role` vẫn chạy nhánh TENANT (tương thích ngược). FE chỉ cần bổ sung `role`/ảnh khi đăng ký landlord.
- OAuth (`loginWithOAuth`) vẫn tạo Tenant ACTIVE — không ảnh hưởng.
- Ảnh CCCD nhạy cảm: lưu local `uploads/landlords/{id}/` chỉ là tạm; cần che static + chuyển S3 (presigned URL) trước production. Lưu **key tương đối** trong DB để chuyển S3 không phải migrate dữ liệu.
- File mồ côi nếu transaction DB fail sau khi đã ghi ảnh: gọi `idCardStorage.remove(landlordId)` để dọn.
- Phải sinh `user_id` ở app (mục 5) thay vì để DB default → kiểm tra không phá vỡ chỗ nào khác đang dựa vào `gen_random_uuid()`.

## Thứ tự thực hiện đề xuất

1. Migration thêm cột duyệt `landlords` (+ backfill `APPROVED` cho landlord cũ).
2. Tách schema `registerTenantSchema` / `registerLandlordSchema` + middleware `validateRegisterByRole`.
3. `uploadIdCards` middleware + map lỗi MulterError → 400.
4. `idCardStorage` (driver `local` ghi `uploads/landlords/{id}/1.jpg`,`2.jpg`) + sửa `createUserWithRole` (nhận `userId` + URL/key ảnh).
5. Service `registerLandlord` (sinh `user_id`, lưu ảnh, rollback xóa file khi fail) + controller `register` rẽ nhánh theo role (giữ route `/register`).
6. Middleware `requireApprovedLandlord` + áp lên route host; thêm `approvalStatus` vào `/me`.
7. Admin: repo + service + route duyệt/từ chối + endpoint stream ảnh CCCD + ghi log + (tùy chọn) thông báo.
8. Test: đăng ký 2 vai trò, verify OTP, chặn host khi PENDING, Admin duyệt → host hoạt động.
9. (Sau) Bổ sung driver `s3` cho `idCardStorage` + chuyển `STORAGE_DRIVER=s3` — không đụng service/controller.
