# 1.14. Tách ảnh CCCD khỏi `POST /register` (đăng ký gọn, upload CCCD ở API riêng)

## Mục tiêu

- `POST /api/auth/register` chỉ nhận **thông tin** (JSON) cho **cả TENANT lẫn LANDLORD** —
  giống hệt nhánh tenant hiện tại. Không còn upload ảnh ở bước đăng ký.
- Việc upload **ảnh mặt trước + sau CCCD** của landlord chuyển sang **một API riêng**.
- Landlord vẫn ở `approval_status = PENDING` chờ Admin duyệt; chỉ khác là ảnh được nộp sau.

## Hiện trạng (rẽ nhánh ở 3 tầng)

| Tầng | Chỗ rẽ nhánh TENANT/LANDLORD |
| --- | --- |
| Route `auth.route.js:24` | `uploadIdCards` (multer.fields) + `validateRegisterByRole` chọn schema theo `role` |
| Controller `auth.controller.js:26-50` | `if (role === 'LANDLORD')` → đọc `req.files`, gọi `registerLandlord` |
| Service `auth.service.js` | `register` (tenant) vs `registerLandlord` (lưu ảnh trước transaction + tạo landlord) |

- `createUserWithRole` (`auth.repository.js:105-114`) đã tự tạo bản ghi `landlords`
  (`approval_status='PENDING'`) hay `tenants` theo `roleName` → **đăng ký landlord không kèm ảnh
  vẫn chạy được** ở tầng repo, chỉ cần ngừng truyền `idCardFrontUrl/BackUrl`.
- **Rủi ro cột NOT NULL + verified host**: `landlords.id_card_front_url/back_url` là
  `notNullable()` (`005_create_landlords.js:14-15`). Code hiện chèn `'' ` khi thiếu
  (`auth.repository.js:108-109`). Nhưng `assertVerifiedHost` (`models/Room.js:285-287`) chỉ kiểm
  `whereNotNull(...)` → **chuỗi rỗng `''` vẫn lọt** ⇒ landlord chưa nộp CCCD bị coi là "đã xác thực
  host". Phải xử lý (xem mục 2 + 5).

## Quyết định thiết kế

1. **Gộp 1 schema đăng ký** thay cho `registerTenantSchema`/`registerLandlordSchema`:
   `role: z.enum(['TENANT','LANDLORD']).default('TENANT')`. Bỏ `validateRegisterByRole` →
   dùng lại `validate({ body: registerSchema })` thường. Hai schema hiện chỉ khác mỗi `role` literal
   nên gộp không mất ràng buộc nào.
2. **Một đường service duy nhất**: `authService.register({ ...fields, role })` cho cả 2 vai trò.
   **Xóa `registerLandlord`** (toàn bộ phần lưu ảnh + sinh `user_id` ở app + rollback file). Landlord
   giờ để DB sinh `user_id` như tenant (không cần biết trước thư mục ảnh nữa).
3. **Cột id_card thành NULLABLE** + chèn `NULL` (không phải `''`) khi đăng ký. Nhờ đó
   `assertVerifiedHost`/`requireApprovedLandlord` phản ánh đúng "chưa nộp CCCD".
4. **API mới** để landlord tự nộp CCCD (yêu cầu đăng nhập, đúng chủ tài khoản). Dùng lại
   `uploadIdCards` (multer) + `idCardStorage.save` + cập nhật 2 cột. Cho phép nộp lại (ghi đè).
5. **Chặn duyệt khi thiếu CCCD**: `approveLandlord` từ chối nếu chưa có đủ 2 ảnh (tránh duyệt hồ sơ rỗng).

## API mới: nộp/cập nhật CCCD landlord

- **Route**: `POST /api/auth/landlord/id-cards`
  `requireAuth` → `uploadIdCards` (multer.fields front/back) → controller.
  (Đặt trong `auth.route.js` cho gần luồng đăng ký; controller mới `submitIdCards` trong `auth.controller.js`.)
- **Controller** `submitIdCards(req,res)`:
  - Lấy `front=req.files?.id_card_front?.[0]`, `back=req.files?.id_card_back?.[0]`.
  - Thiếu ảnh → `AppError('ID_CARD_REQUIRED', ..., 400)` (giữ thông điệp cũ).
  - Gọi `authService.submitLandlordIdCards({ userId: req.user.userId, front, back })`.
- **Service** `submitLandlordIdCards({ userId, front, back })`:
  - Kiểm user là LANDLORD (qua `findUserById` → `role_name`), nếu không → `403`.
  - (Khuyến nghị) chỉ cho nộp khi `approval_status ∈ {PENDING, REJECTED}` — đã APPROVED thì không cần.
  - `idCardStorage.save({ landlordId: userId, frontFile: front, backFile: back })` (ghi đè tên cố định 1/2).
  - `authRepository.updateLandlordIdCards(userId, { frontKey, backKey })`.
  - Nộp lại sau khi bị REJECTED: reset `approval_status='PENDING'`, `rejection_reason=null` (tùy nghiệp vụ — nêu rõ để quyết).
- **Repository** `updateLandlordIdCards(landlordId, { frontKey, backKey })`:
  `db('landlords').where({landlord_id}).update({ id_card_front_url: frontKey, id_card_back_url: backKey })`.

> Lựa chọn thời điểm nộp: **yêu cầu đăng nhập** (đã verify OTP → login) cho đơn giản & an toàn
> (chủ tài khoản tự nộp). Nếu muốn nộp ngay trong lúc còn INACTIVE (chưa verify OTP) thì cần endpoint
> không-auth định danh bằng email + token tạm — phức tạp hơn, **không khuyến nghị** cho phạm vi này.

## Luồng sau refactor

1. `POST /register {..., role:'LANDLORD'}` (JSON) → tạo user INACTIVE + `landlords(PENDING, id_card=NULL)` → gửi OTP.
2. `POST /verify-otp` → ACTIVE (vẫn PENDING duyệt).
3. `POST /login` → có token.
4. `POST /auth/landlord/id-cards` (multipart) → lưu ảnh + cập nhật 2 cột.
5. Admin `getLandlordIdCard` xem ảnh → `approveLandlord` (chặn nếu thiếu ảnh) / `rejectLandlord`.

## Kiểm thử

| Case | Kỳ vọng |
| --- | --- |
| Đăng ký TENANT (JSON) | 201, tạo tenant — không đổi hành vi |
| Đăng ký LANDLORD (JSON, không ảnh) | 201, `landlords` PENDING, `id_card_*_url = NULL`; message nhắc chờ duyệt |
| Đăng ký LANDLORD gửi kèm multipart cũ | Vẫn 201 (ảnh bị bỏ qua); FE cần cập nhật gọi API mới |
| Nộp CCCD khi đã đăng nhập (landlord) | 200, 2 cột có key; xem được qua admin endpoint |
| Nộp CCCD khi chưa đăng nhập | 401 |
| Tenant gọi API nộp CCCD | 403 |
| Nộp thiếu 1 ảnh | 400 `ID_CARD_REQUIRED` |
| Admin duyệt landlord chưa nộp CCCD | 400 (chặn), không chuyển APPROVED |
| `assertVerifiedHost` với landlord chưa nộp | Trả "chưa xác thực" (vì NULL, không còn `''` lọt) |
| Nộp lại sau REJECTED | Ghi đè ảnh; (nếu chọn) về PENDING |

## Ghi chú / rủi ro

- **Đồng bộ tài liệu 1.11 (S3)**: 1.11 đổi ruột `idCardStorage` sang S3 nhưng **vẫn giữ interface
  `save/remove`** → API nộp CCCD mới dùng lại không đổi. Hai refactor độc lập, không xung đột.
- **Frontend phải đổi**: đã được chỉnh và hoàn thiện như kế hoạch này
- **Dữ liệu cũ**: landlord đã đăng ký bằng luồng cũ có ảnh `''`/key thật vẫn hợp lệ (migration chỉ nới
  NULL, không xóa dữ liệu). Backfill không cần.
- **Bỏ rollback file trong register**: vì không còn ghi ảnh trước transaction nên không cần dọn rác —
  giảm độ phức tạp so với `registerLandlord` cũ.
- Cân nhắc giữ alias `registerLandlordSchema` để không vỡ import khác (rà `grep` trước khi xóa).
