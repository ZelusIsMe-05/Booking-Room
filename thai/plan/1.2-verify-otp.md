# 1.2. Xác thực OTP đăng ký

## Mục tiêu

Xây dựng API xác thực OTP cho tài khoản mới đăng ký. Nếu OTP hợp lệ, tài khoản chuyển từ `INACTIVE` sang `ACTIVE`.

## Luồng hoạt động

1. Client gửi email kèm OTP.
2. Backend tìm user đang `INACTIVE` theo email. `identifier` = email đã normalize (trim + lowercase).
3. Backend đọc OTP từ Redis: `HGETALL otp:REGISTRATION:{email}`.
4. Nếu key không còn (đã hết hạn theo TTL hoặc chưa từng có) → coi như **OTP hết hạn** (`410`).
5. Backend kiểm tra `attempts < OTP_MAX_ATTEMPTS` (mặc định 3). Nếu đã đạt giới hạn → `429`.
6. Backend so khớp `otp` client gửi với `code` (plaintext) trong Redis.
7. Nếu OTP sai → `HINCRBY otp:REGISTRATION:{email} attempts 1`; trả `remainingAttempts = max - attempts`. Khi chạm giới hạn có thể `DEL` key.
8. Nếu OTP đúng → `DEL otp:REGISTRATION:{email}` (consume) và cập nhật user `status = ACTIVE` (+ `verified_at = now` nếu có cột).
9. Backend trả response xác thực thành công.

> Redis không phân biệt được "OTP hết hạn" và "OTP chưa từng tồn tại" (cả hai đều là key không có). Quy ước: user `INACTIVE` + không có key ⇒ trả `410` "hết hạn, yêu cầu gửi lại". Chỉ trả `404` khi không tìm thấy user `INACTIVE` phù hợp.

## API

| Thuộc tính | Giá trị |
| --- | --- |
| Method | `POST` |
| Path | `/api/auth/verify-otp` |
| Auth required | Không |
| Content-Type | `application/json` |

## Header

```http
Content-Type: application/json
Accept: application/json
```

## Request body

```json
{
  "email": "vana@example.com",
  "otp": "123456"
}
```

## Request schema

| Field | Type | Required | Rule |
| --- | --- | --- | --- |
| `email` | string | Có | Đúng định dạng email |
| `otp` | string | Có | 6 chữ số |

> OTP chỉ gửi qua email (không có SMS), nên verify chỉ nhận `email`.

## HTTP code

| Code | Trường hợp |
| --- | --- |
| `200` | OTP hợp lệ, tài khoản được kích hoạt |
| `400` | Request thiếu dữ liệu hoặc OTP sai |
| `404` | Không tìm thấy tài khoản/OTP phù hợp |
| `410` | OTP hết hạn |
| `429` | OTP đã nhập sai quá 3 lần |
| `500` | Lỗi server |

## Response thành công

```json
{
  "success": true,
  "message": "Xác thực OTP thành công. Tài khoản đã được kích hoạt.",
  "data": {
    "userId": "9d971ee5-7e97-4d6c-b70b-d9a7a91b5a00",
    "status": "ACTIVE"
  }
}
```

## Response OTP sai

```json
{
  "success": false,
  "message": "Mã OTP không chính xác.",
  "data": {
    "remainingAttempts": 2
  }
}
```

## Response OTP hết hạn

```json
{
  "success": false,
  "message": "Mã OTP đã hết hạn. Vui lòng yêu cầu gửi lại mã mới."
}
```

## Response quá số lần nhập sai

```json
{
  "success": false,
  "message": "Mã OTP đã bị khóa do nhập sai quá số lần cho phép. Vui lòng yêu cầu gửi lại mã mới."
}
```

## Response schema

| Field | Type | Ghi chú |
| --- | --- | --- |
| `success` | boolean | Trạng thái request |
| `message` | string | Thông báo tổng quát |
| `data.userId` | uuid | ID user được kích hoạt |
| `data.status` | string | `ACTIVE` sau khi OTP đúng |
| `data.remainingAttempts` | number | Chỉ trả khi OTP sai |

## Bảng liên quan

### `users`

Đọc và cập nhật:

- Tìm theo `email`.
- Chỉ xử lý user `status = INACTIVE`.
- Cập nhật `status = ACTIVE` khi OTP đúng.

### OTP — Redis (thay cho `otp_verifications`)

Key: `otp:REGISTRATION:{email}` (email đã normalize). Value Hash: `{ code, attempts }`.

Đọc và cập nhật:

- `HGETALL` để lấy `code` + `attempts`. Key không có ⇒ OTP hết hạn (`410`).
- So khớp `code` (so sánh timing-safe); sai → `HINCRBY ... attempts 1` (atomic, không reset TTL).
- `attempts >= OTP_MAX_ATTEMPTS` ⇒ `429`.
- OTP đúng → `DEL` key (consume, dùng một lần) và set user `ACTIVE`.

> Gợi ý cứng hơn (chống race khi double-submit): gộp bước "đọc + so khớp + tăng attempts" vào một **Lua script** chạy atomic trên Redis. MVP có thể dùng `HGETALL` + `HINCRBY` tuần tự.

## Điều kiện cần để thực hiện

1. User đã đăng ký và đang `INACTIVE`.
2. Còn key OTP trong Redis (chưa quá TTL 5 phút).
3. `attempts < OTP_MAX_ATTEMPTS` (mặc định 3).
4. Redis đang chạy và `REDIS_URL` đã cấu hình.

## Kết quả mong đợi

1. OTP đúng kích hoạt tài khoản.
2. OTP sai không kích hoạt tài khoản.
3. OTP hết hạn bị từ chối.
4. OTP nhập sai quá 3 lần bị từ chối và yêu cầu gửi lại.
5. OTP đã consumed không dùng lại được.

## Ghi chú triển khai

- Chống double-submit kích hoạt nhiều lần: **`DEL` key OTP trước** rồi mới set user `ACTIVE`. `DEL` trả về `1` đúng một lần — chỉ request "thắng" mới tiếp tục kích hoạt; các request sau thấy key đã mất ⇒ `410`.
- Không trả về `code` OTP trong bất kỳ response nào.
- Không cấp JWT tự động ở bước này trừ khi sản phẩm chốt auto-login.
