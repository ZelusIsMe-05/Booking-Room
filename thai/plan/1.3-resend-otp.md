# 1.3. Gửi lại OTP

## Mục tiêu

Xây dựng API gửi lại OTP đăng ký cho user chưa kích hoạt tài khoản.

## Luồng hoạt động

1. Client gửi email.
2. Backend tìm user đang `INACTIVE` theo email. Nếu user đã `ACTIVE` → `409`. `identifier` = email đã normalize.
3. Backend kiểm tra rate-limit: nếu key cooldown `otp:cooldown:REGISTRATION:{email}` còn tồn tại → `429`.
4. Backend sinh OTP mới.
5. Backend **ghi đè** Redis: `HSET otp:REGISTRATION:{email} code <mới> attempts 0` + `EXPIRE 300`. Ghi đè tự nhiên làm OTP cũ vô hiệu, không cần bước dọn riêng.
6. Backend đặt cooldown: `SET otp:cooldown:REGISTRATION:{email} 1 EX OTP_RESEND_COOLDOWN_SECONDS` (mặc định 60s).
7. Backend gửi OTP mới qua email.
8. Backend trả response thành công.

## API

| Thuộc tính | Giá trị |
| --- | --- |
| Method | `POST` |
| Path | `/api/auth/resend-otp` |
| Auth required | Không |
| Content-Type | `application/json` |

## Header

```http
Content-Type: application/json
Accept: application/json
```

## Request body

```json
{
  "email": "vana@example.com",
  "purpose": "REGISTRATION"
}
```

## Request schema

| Field | Type | Required | Rule |
| --- | --- | --- | --- |
| `email` | string | Có | Đúng định dạng email |
| `purpose` | string | Có | MVP chỉ nhận `REGISTRATION` |

> OTP chỉ gửi qua email (không có SMS), nên resend chỉ nhận `email`.

## HTTP code

| Code | Trường hợp |
| --- | --- |
| `200` | OTP mới đã được gửi |
| `400` | Request không hợp lệ |
| `404` | Không tìm thấy tài khoản đang chờ xác thực |
| `409` | Tài khoản đã active, không cần gửi OTP đăng ký |
| `429` | Gửi lại quá thường xuyên nếu có rate limit |
| `500` | Lỗi server hoặc lỗi gửi email |

## Response thành công

```json
{
  "success": true,
  "message": "Mã OTP mới đã được gửi.",
  "data": {
    "otpExpiresInSeconds": 300
  }
}
```

## Response tài khoản đã active

```json
{
  "success": false,
  "message": "Tài khoản đã được kích hoạt."
}
```

## Response schema

| Field | Type | Ghi chú |
| --- | --- | --- |
| `success` | boolean | Trạng thái request |
| `message` | string | Thông báo tổng quát |
| `data.otpExpiresInSeconds` | number | `300` giây |

## Bảng liên quan

### `users`

Đọc:

- Tìm theo `email`.
- Chỉ cho resend nếu `status = INACTIVE`.

### OTP — Redis (thay cho `otp_verifications`)

Ghi/cập nhật:

- `HSET otp:REGISTRATION:{email} code <mới> attempts 0` + `EXPIRE 300` → ghi đè OTP cũ (reset cả `attempts` và TTL).
- `SET otp:cooldown:REGISTRATION:{email} 1 EX <cooldown>` để chặn resend dồn dập.

## Điều kiện cần để thực hiện

1. User tồn tại.
2. User đang `INACTIVE`.
3. Có cấu hình gửi email OTP.
4. Redis đang chạy và `REDIS_URL` đã cấu hình.
5. Chưa trong thời gian cooldown (`OTP_RESEND_COOLDOWN_SECONDS`, mặc định 60s).

## Kết quả mong đợi

1. OTP cũ không còn dùng được (bị ghi đè trong Redis).
2. OTP mới có hạn 5 phút (TTL Redis), `attempts` reset về 0.
3. OTP mới được gửi đến email đã đăng ký.
4. Không trả OTP raw trong response.
5. Resend trong thời gian cooldown bị từ chối `429`.

## Ghi chú triển khai

- Rate limit resend bằng key cooldown Redis (`otp:cooldown:REGISTRATION:{email}`, TTL = `OTP_RESEND_COOLDOWN_SECONDS`) — đơn giản, tự hết hạn, không cần bảng riêng.
- `identifier` (email) phải normalize giống `1.1`/`1.2` để key trùng đúng.
- Không trả `code` OTP trong response.
