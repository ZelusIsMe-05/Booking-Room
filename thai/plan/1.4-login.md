# 1.4. Đăng nhập

## Mục tiêu

Xây dựng API đăng nhập bằng email/số điện thoại/username và mật khẩu. Nếu thông tin hợp lệ, backend cấp một cặp JWT gồm **access token** (ngắn hạn) và **refresh token** (dài hạn) kèm thông tin user.

## Luồng hoạt động

1. Client gửi `identifier` và `password`.
2. Backend tìm user theo email, số điện thoại hoặc username.
3. Backend ghi audit failed nếu không tìm thấy user.
4. Backend kiểm tra tài khoản có đang bị khóa không.
5. Backend kiểm tra trạng thái user.
6. Backend so sánh mật khẩu bằng bcrypt.
7. Nếu sai mật khẩu, tăng số lần đăng nhập sai.
8. Nếu sai quá 5 lần liên tiếp, khóa tài khoản 10 phút.
9. Nếu đúng mật khẩu, reset số lần sai và cập nhật `last_login_at`.
10. Backend ghi audit success.
11. Backend tạo cặp access token + refresh token. Refresh token mang `jti` (id phiên), backend **lưu phiên refresh vào bảng `refresh_tokens`** (lưu hash của token, không lưu token thô) để sau này có thể thu hồi khi logout.
12. Backend trả `{ tokenType, accessToken, accessExpiresIn, refreshToken, refreshExpiresIn, user }`.

## API

| Thuộc tính | Giá trị |
| --- | --- |
| Method | `POST` |
| Path | `/api/auth/login` |
| Auth required | Không |
| Content-Type | `application/json` |

## Header

```http
Content-Type: application/json
Accept: application/json
User-Agent: Mozilla/5.0
```

Backend cần đọc IP từ request:

- `req.ip`
- hoặc `X-Forwarded-For` nếu chạy sau proxy tin cậy

## Request body

```json
{
  "identifier": "vana@example.com",
  "password": "Password@123"
}
```

## Request schema

| Field | Type | Required | Rule |
| --- | --- | --- | --- |
| `identifier` | string | Có | Email, số điện thoại hoặc username |
| `password` | string | Có | Không rỗng |

## HTTP code

| Code | Trường hợp |
| --- | --- |
| `200` | Đăng nhập thành công |
| `400` | Thiếu identifier/password |
| `401` | Sai tài khoản hoặc mật khẩu |
| `403` | Tài khoản chưa active hoặc bị banned |
| `423` | Tài khoản đang bị khóa tạm thời |
| `500` | Lỗi server |

## Response thành công

```json
{
  "success": true,
  "message": "Đăng nhập thành công.",
  "data": {
    "tokenType": "Bearer",
    "accessToken": "jwt-access-token",
    "accessExpiresIn": "15m",
    "refreshToken": "jwt-refresh-token",
    "refreshExpiresIn": "7d",
    "user": {
      "userId": "9d971ee5-7e97-4d6c-b70b-d9a7a91b5a00",
      "fullName": "Nguyen Van A",
      "email": "vana@example.com",
      "phoneNumber": "0912345678",
      "username": "vana",
      "role": "TENANT",
      "status": "ACTIVE"
    }
  }
}
```

## Response sai thông tin

```json
{
  "success": false,
  "message": "Tài khoản hoặc mật khẩu không chính xác."
}
```

## Response tài khoản bị khóa

```json
{
  "success": false,
  "message": "Tài khoản đang bị khóa tạm thời. Vui lòng thử lại sau.",
  "data": {
    "lockedUntil": "2026-06-13T05:30:00.000Z"
  }
}
```

## Response tài khoản chưa active

```json
{
  "success": false,
  "message": "Tài khoản chưa được xác thực OTP."
}
```

## Response schema

| Field | Type | Ghi chú |
| --- | --- | --- |
| `success` | boolean | Trạng thái request |
| `message` | string | Thông báo tổng quát |
| `data.tokenType` | string | `Bearer` |
| `data.accessToken` | string | JWT access token (ngắn hạn) |
| `data.accessExpiresIn` | string | Theo `JWT_ACCESS_EXPIRES_IN` |
| `data.refreshToken` | string | JWT refresh token (dài hạn) |
| `data.refreshExpiresIn` | string | Theo `JWT_REFRESH_EXPIRES_IN` |
| `data.user` | object | User public data |
| `data.lockedUntil` | datetime | Chỉ có khi tài khoản bị khóa (trong nhánh lỗi `423`) |

## JWT payload

Access token mang định danh + ngữ cảnh phân quyền để các module khác rút UUID từ `sub`:

```json
{
  "sub": "9d971ee5-7e97-4d6c-b70b-d9a7a91b5a00",
  "role": "TENANT",
  "status": "ACTIVE",
  "type": "access"
}
```

Refresh token tối giản, mang thêm `jti` (id phiên) để đối chiếu với bản ghi trong DB:

```json
{
  "sub": "9d971ee5-7e97-4d6c-b70b-d9a7a91b5a00",
  "type": "refresh",
  "jti": "f1c2a0d4-1b2c-4e5f-8a9b-0c1d2e3f4a5b"
}
```

Hai token ký bằng secret riêng (`JWT_ACCESS_SECRET`, `JWT_REFRESH_SECRET`). Access token chỉ gửi kèm ở các request tài nguyên; refresh token chỉ gửi tới `POST /api/auth/refresh`.

### Mô hình token: access ngắn hạn + refresh có trạng thái (DB-backed)

- **Access token**: TTL ngắn (mặc định `15m`), **stateless** — không lưu DB, chỉ verify chữ ký + hạn. Đây là thứ gửi kèm mọi request tài nguyên.
- **Refresh token**: TTL dài (mặc định `7d`), **có trạng thái** — mỗi phiên được lưu một bản ghi trong bảng `refresh_tokens` (lưu `token_hash`, không lưu token thô).
- Khi `POST /api/auth/refresh`: backend verify chữ ký + hạn, rồi **tra `jti` trong `refresh_tokens`**. Nếu bản ghi không còn (đã logout) → từ chối, không cấp access token mới.
- Khi `POST /api/auth/logout`: backend **xóa bản ghi refresh token** khỏi DB. Từ đó refresh token cũ không dùng được nữa; access token cũ vẫn hợp lệ tối đa tới khi hết hạn (≤ 15 phút) rồi không gia hạn được → user bị đăng xuất hoàn toàn.

> Đánh đổi: không cần access-token blacklist, cửa sổ rủi ro nhỏ (≤ TTL access). Nhược điểm: trong cửa sổ đó access token cũ vẫn dùng được. Đây **không phải rotation đầy đủ** (refresh token được tái dùng tới khi hết hạn hoặc bị thu hồi); rotation theo từng lần refresh là bước nâng cấp tùy chọn ở phase sau — xem `1.6-logout.md`.

## Bảng liên quan

### `users`

Đọc:

- Tìm theo `email`, `phone_number`, hoặc `username`.
- Lấy `password`, `status`, `role_id`.

### `roles`

Đọc:

- Join để lấy `role_name`.

### `account_security`

Đọc/cập nhật:

- `failed_login_attempts`
- `locked_until`
- `last_login_at`
- `updated_at`

### `login_audit_logs`

Ghi mới cho mọi lần login:

- `user_id`
- `login_identifier`
- `success`
- `failure_reason`
- `ip_address`
- `user_agent`
- `created_at`

### `refresh_tokens`

Ghi mới mỗi lần đăng nhập thành công (một phiên = một bản ghi):

- `token_id` (UUID, đồng thời là `jti` trong refresh token)
- `user_id`
- `token_hash` (SHA-256 của refresh token; **không** lưu token thô)
- `user_agent`, `ip_address` (tùy chọn, để quản lý phiên)
- `expires_at` (khớp `JWT_REFRESH_EXPIRES_IN`)
- `created_at`

> Bảng này **chưa có** trong các migration hiện tại — cần thêm migration mới (`021_create_refresh_tokens.js`). Chi tiết cột xem `1.6-logout.md`.

## Điều kiện cần để thực hiện

1. User đã tồn tại.
2. User phải `ACTIVE` để đăng nhập thành công.
3. User không bị `BANNED`.
4. Tài khoản không đang bị khóa tạm thời.
5. `JWT_ACCESS_SECRET` và `JWT_REFRESH_SECRET` đã cấu hình.
6. `JWT_ACCESS_EXPIRES_IN` (default `15m`) và `JWT_REFRESH_EXPIRES_IN` (default `7d`) đã cấu hình.
7. Bảng `refresh_tokens` đã tồn tại (migration `021_create_refresh_tokens.js`).

## Kết quả mong đợi

1. Đăng nhập đúng trả về cặp access token + refresh token.
2. Đăng nhập sai không tiết lộ tài khoản có tồn tại hay không.
3. Sai quá 5 lần liên tiếp khóa tài khoản 10 phút.
4. Đăng nhập đúng reset số lần sai.
5. Mọi lần đăng nhập đều có audit log.
6. Response không trả password hash.
7. Mỗi lần đăng nhập thành công tạo một bản ghi phiên trong `refresh_tokens` (lưu hash), phục vụ thu hồi khi logout.

## Ghi chú triển khai

- Lỗi sai user và sai password đều dùng cùng message chung.
- Nên ghi audit log cả khi user không tồn tại, với `user_id = null`.
- Cần cẩn thận timezone khi tính `locked_until`.
