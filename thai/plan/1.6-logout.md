# 1.6. Đăng xuất

## Mục tiêu

Xây dựng API đăng xuất cho mô hình JWT stateless. Backend không cần hủy token ở MVP; client chịu trách nhiệm xóa token đang lưu.

## Luồng hoạt động

1. Client gửi request logout.
2. Nếu có token, middleware có thể verify token để biết user đang logout.
3. Backend trả response thành công.
4. Client xóa JWT khỏi nơi lưu trữ.

## API

| Thuộc tính | Giá trị |
| --- | --- |
| Method | `POST` |
| Path | `/api/auth/logout` |
| Auth required | Khuyến nghị có, nhưng MVP có thể cho phép gọi idempotent |
| Content-Type | `application/json` |

## Header

```http
Accept: application/json
Authorization: Bearer <jwt-token>
```

Nếu triển khai idempotent logout, `Authorization` có thể optional.

## Request body

Không có.

## HTTP code

| Code | Trường hợp |
| --- | --- |
| `200` | Đăng xuất thành công |
| `401` | Token sai nếu endpoint bắt buộc auth |
| `500` | Lỗi server |

## Response thành công

```json
{
  "success": true,
  "message": "Đăng xuất thành công."
}
```

## Response schema

| Field | Type | Ghi chú |
| --- | --- | --- |
| `success` | boolean | Trạng thái request |
| `message` | string | Thông báo tổng quát |

## Bảng liên quan

Không bắt buộc ghi DB trong MVP.

Nếu muốn audit logout ở phase sau, có thể ghi vào:

- `system_logs`

## Điều kiện cần để thực hiện

1. Dự án dùng JWT stateless.
2. Client có cơ chế xóa token sau response.

## Kết quả mong đợi

1. API trả success ổn định.
2. Client xóa token và không còn gọi API protected bằng token cũ.
3. Backend không phải lưu session/token blacklist trong MVP.

## Ghi chú triển khai

- Token cũ vẫn hợp lệ cho đến khi hết hạn nếu không có blacklist. Đây là giới hạn của JWT stateless.
- Nếu sản phẩm yêu cầu thu hồi token ngay lập tức, cần bổ sung token blacklist hoặc session table ở phase sau.
