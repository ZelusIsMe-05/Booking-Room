# 1.6. Đăng xuất

## Mục tiêu

Xây dựng API đăng xuất theo mô hình **access token ngắn hạn (stateless) + refresh token có trạng thái trong DB**. Đăng xuất = **xóa bản ghi refresh token** của phiên khỏi bảng `refresh_tokens`. Sau đó refresh token cũ không thể gia hạn access token; access token cũ tự hết hạn trong tối đa TTL của nó (mặc định 15 phút) → user bị đăng xuất hoàn toàn.

## Bối cảnh mô hình token

- **Access token**: TTL ngắn (`JWT_ACCESS_EXPIRES_IN`, mặc định `15m`), stateless, gửi kèm mọi request tài nguyên. Không lưu DB, không thu hồi trực tiếp.
- **Refresh token**: TTL dài (`JWT_REFRESH_EXPIRES_IN`, mặc định `7d`), mỗi phiên có một bản ghi trong `refresh_tokens` (lưu `token_hash`). Chỉ gửi tới `/api/auth/refresh` và `/api/auth/logout`.
- **Refresh** (`POST /api/auth/refresh`) phải đổi hành vi: ngoài verify chữ ký + hạn, còn **tra `jti` trong `refresh_tokens`**; nếu không còn bản ghi (đã logout) thì từ chối.

> Đây không phải rotation đầy đủ: refresh token được tái dùng tới khi hết hạn hoặc bị thu hồi. Rotation theo từng lần refresh là nâng cấp tùy chọn (xem Ghi chú triển khai).

## Luồng hoạt động

1. Client gửi `POST /api/auth/logout` kèm access token (Authorization) và `refreshToken` trong body.
2. `requireAuth` verify access token → lấy `userId`.
3. Backend verify refresh token (best-effort) để lấy `jti`. Nếu refresh token sai định dạng/sai chữ ký thì vẫn xử lý idempotent (xem bên dưới).
4. Backend **xóa bản ghi** `refresh_tokens` theo `token_id = jti` **và** `user_id = userId` (chỉ chủ sở hữu mới xóa được phiên của mình).
5. Dù bản ghi có tồn tại hay không, trả `200` (idempotent) để tránh lộ thông tin và để client gọi lại an toàn.
6. Client xóa access token + refresh token đang lưu.

## API

| Thuộc tính | Giá trị |
| --- | --- |
| Method | `POST` |
| Path | `/api/auth/logout` |
| Auth required | Có (`requireAuth` bằng access token) |
| Content-Type | `application/json` |

## Header

```http
Authorization: Bearer <access-token>
Content-Type: application/json
Accept: application/json
```

## Request body

```json
{
  "refreshToken": "jwt-refresh-token"
}
```

## Request schema

| Field | Type | Required | Rule |
| --- | --- | --- | --- |
| `refreshToken` | string | Có | Không rỗng. Refresh token của phiên cần đăng xuất |

> Validate bằng `validate({ body: logoutSchema })` (Zod) như convention — xem `GUILD/convention-model-validate.md`. `logoutSchema` đặt trong `models/User.js` cạnh `loginSchema`, `refreshSchema`.

## HTTP code

| Code | Trường hợp |
| --- | --- |
| `200` | Đăng xuất thành công (kể cả khi phiên đã bị xóa trước đó — idempotent) |
| `400` | Thiếu `refreshToken` |
| `401` | Thiếu/sai access token (không qua `requireAuth`) |
| `500` | Lỗi server |

## Thay đổi kéo theo ở `POST /api/auth/refresh`

Sau khi có bảng `refresh_tokens`, luồng refresh phải:

1. Verify refresh token (chữ ký + hạn + `type = refresh`).
2. Lấy `jti` và `sub`.
3. `SELECT` `refresh_tokens` theo `token_id = jti`.
   - Không tìm thấy → `401` (đã logout hoặc token giả).
   - (Khuyến nghị) so khớp `token_hash` với SHA-256 của token nhận được → phòng trường hợp secret rò rỉ.
   - `expires_at` đã qua → `401`, đồng thời xóa bản ghi rác.
4. Tải user theo `sub`, kiểm tra `ACTIVE`.
5. Cấp access token mới (refresh token giữ nguyên).

## Bảng liên quan

### `refresh_tokens` (bảng mới — cần migration)

Logout: **xóa** bản ghi theo `token_id` + `user_id`.

| Cột | Kiểu | Ràng buộc |
| --- | --- | --- |
| `token_id` | UUID | PRIMARY KEY. Chính là `jti` của refresh token (app sinh bằng `crypto.randomUUID()`) |
| `user_id` | UUID | NOT NULL, FK → `users(user_id)` ON DELETE CASCADE |
| `token_hash` | VARCHAR(255) | NOT NULL. SHA-256 (hex) của refresh token. **Không** lưu token thô |
| `user_agent` | VARCHAR(512) | NULL. Phục vụ quản lý phiên/đa thiết bị |
| `ip_address` | INET | NULL |
| `expires_at` | TIMESTAMP | NOT NULL. Khớp TTL refresh token |
| `created_at` | TIMESTAMP | NOT NULL, DEFAULT `now()` |
| `revoked_at` | TIMESTAMP | NULL. Chỉ dùng nếu chọn phương án soft-revoke thay vì xóa cứng (xem Ghi chú) |

Index gợi ý:

- `refresh_tokens_user_id_idx` trên `(user_id)` — để logout-all và liệt kê phiên.
- `refresh_tokens_expires_at_idx` trên `(expires_at)` — để job dọn token hết hạn.

### `system_logs` (tùy chọn)

Có thể ghi audit logout ở phase sau (`user_id`, `ip_address`, `user_agent`, thời điểm).

- `utils/jwt.js`: `signRefreshToken` thêm `jwtid` (jti); bổ sung helper lấy `jti` từ payload nếu cần.
- `repositories/auth/auth.repository.js`: thêm `insertRefreshToken`, `findRefreshTokenById`, `deleteRefreshToken`, (tùy chọn) `deleteAllRefreshTokensForUser`.
- `services/auth/auth.service.js`:
  - `login`/`issueTokens`: sinh `jti`, ký refresh token, `INSERT` `refresh_tokens` (lưu hash).
  - `refreshAccessToken`: thêm bước tra DB theo `jti`.
  - thêm `logout({ userId, refreshToken })`: xóa bản ghi, idempotent.
- `controllers/auth/auth.controller.js`: thêm `logout`.
- `routes/auth/auth.route.js`: `router.post('/logout', requireAuth, validate({ body: logoutSchema }), authController.logout)`.

## Điều kiện cần để thực hiện

1. Bảng `refresh_tokens` đã tồn tại (migration `021`).
2. Login đã lưu refresh token vào `refresh_tokens` (xem `1.4-login.md`).
3. Endpoint refresh đã đối chiếu `jti` với DB.
4. `requireAuth` hoạt động (access token hợp lệ).

## Kết quả mong đợi

1. Sau logout, gọi `POST /api/auth/refresh` bằng refresh token cũ trả `401` (không cấp access token mới).
2. Access token cũ vẫn dùng được tới khi hết hạn (≤ TTL access, mặc định 15 phút), sau đó không gia hạn được → user đăng xuất hoàn toàn.
3. Logout là idempotent: gọi lại nhiều lần vẫn trả `200`.
4. Một user không thể xóa phiên của user khác (ràng buộc `user_id` trong điều kiện xóa).
5. Không cần access-token blacklist; cửa sổ rủi ro ≤ TTL access.

## Ghi chú triển khai

- **Idempotent**: nếu `refreshToken` sai chữ ký/hết hạn/không còn trong DB, vẫn trả `200`. Mục tiêu logout là "đảm bảo phiên không còn", không phải xác minh token.
- **Bảo mật**: lưu `token_hash` (SHA-256) thay vì token thô; nếu DB rò rỉ, kẻ tấn công không có sẵn refresh token dùng được.
- **Logout tất cả thiết bị** (biến thể): thêm cờ `allDevices` hoặc endpoint riêng → `DELETE FROM refresh_tokens WHERE user_id = ?`.
- **Soft-revoke** (biến thể): thay vì xóa cứng, set `revoked_at = now()` và lọc `revoked_at IS NULL` khi refresh — giữ lại lịch sử phiên để audit.
- **Dọn rác**: nên có job định kỳ xóa bản ghi `expires_at < now()` (gắn vào `1.7-cleanup-unverified.md` hoặc job riêng).
- **Rotation đầy đủ** (nâng cấp tương lai): mỗi lần refresh thì cấp refresh token mới + xóa cái cũ; nếu phát hiện refresh token đã dùng lại (replay) thì thu hồi toàn bộ phiên của user.
- **Timezone**: tính `expires_at` nhất quán với cách ký JWT (UTC).
