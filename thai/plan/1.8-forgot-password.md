# 1.8. Quên mật khẩu (Forgot password / Reset password)

## Mục tiêu

Cho phép user **đã kích hoạt** (`ACTIVE`) đặt lại mật khẩu khi quên, qua OTP gửi email.
Tái dùng hạ tầng OTP Redis sẵn có (`redis/otpStore.js`) với một `purpose` mới
`PASSWORD_RESET`, tách biệt hoàn toàn với OTP đăng ký (`REGISTRATION`).

Gồm 2 endpoint:

| Bước | Endpoint | Vai trò |
| --- | --- | --- |
| 1 | `POST /api/auth/forgot-password` | Yêu cầu mã OTP đặt lại mật khẩu (gửi qua email) |
| 2 | `POST /api/auth/reset-password` | Gửi OTP + mật khẩu mới → đổi mật khẩu |

Đồng thời **mở rộng `resend-otp`** (mục [§5](#5-mở-rộng-resend-otp-đa-purpose)) để gửi lại OTP cho cả 2 purpose.

---

## 1. `POST /api/auth/forgot-password` — yêu cầu OTP

### Luồng hoạt động

1. Client gửi `email`.
2. Backend tìm user `ACTIVE` theo email (`identifier` = email normalize: trim + lowercase).
3. **Chống dò tài khoản (enumeration):** dù email không tồn tại / chưa active vẫn **trả `200` với message chung**. Chỉ khi user `ACTIVE` mới thực sự sinh và gửi OTP.
4. Nếu còn cooldown `otp:cooldown:PASSWORD_RESET:{email}` → vẫn trả `200` (không lộ trạng thái), nhưng không gửi lại mã.
5. Sinh OTP mới, `setOtp({ purpose: PASSWORD_RESET, identifier: email, code })` (HSET + EXPIRE 300).
6. Đặt cooldown `setResendCooldown({ purpose: PASSWORD_RESET, identifier: email })`.
7. Gửi OTP qua email với `purpose = PASSWORD_RESET`.
8. Trả `200`.

### API

| Thuộc tính | Giá trị |
| --- | --- |
| Method | `POST` |
| Path | `/api/auth/forgot-password` |
| Auth required | Không |
| Content-Type | `application/json` |

### Request body

```json
{ "email": "vana@example.com" }
```

| Field | Type | Required | Rule |
| --- | --- | --- | --- |
| `email` | string | Có | Đúng định dạng email |

### HTTP code

| Code | Trường hợp |
| --- | --- |
| `200` | Đã xử lý yêu cầu (luôn trả 200 nếu request hợp lệ — chống enumeration) |
| `400` | Request không hợp lệ (sai định dạng email) |
| `500` | Lỗi server / lỗi gửi email |

### Response thành công (luôn dùng message chung)

```json
{
  "success": true,
  "message": "Nếu email tồn tại và đã kích hoạt, mã OTP đặt lại mật khẩu đã được gửi.",
  "data": { "otpExpiresInSeconds": 300 }
}
```

> Không tiết lộ email có tồn tại hay không. `data.otpExpiresInSeconds` để client hiển thị đếm ngược; không trả `code` OTP.

## 2. `POST /api/auth/reset-password` — đặt lại mật khẩu

### Luồng hoạt động

1. Client gửi `email`, `otp`, `newPassword`, `confirmPassword`.
2. Backend tìm user `ACTIVE` theo email. Không có → `404` (giai đoạn này user đã chủ động nhập OTP nhận qua mail nên việc báo 404 chấp nhận được).
3. `verifyOtp({ purpose: PASSWORD_RESET, identifier: email, code: otp })`:
   - `EXPIRED` (key không còn) → `410`.
   - `LOCKED` (đã sai ≥ `OTP_MAX_ATTEMPTS`) → `429`.
   - `WRONG` → `incrAttempts`; nếu hết lượt → `429`, còn lượt → `400` + `remainingAttempts`.
   - `OK` → đi tiếp.
4. **`DEL` key OTP trước** (consume, chống double-submit) rồi:
   - Hash `newPassword` (`utils/hashPassword`), cập nhật `users.password`.
   - **Thu hồi toàn bộ refresh token của user** (xóa hết phiên trong `refresh_tokens`) để đăng xuất mọi thiết bị sau khi đổi mật khẩu.
   - Reset `failed_login_attempts`, `locked_until` trong `account_security`.
5. Trả `200`.

### API

| Thuộc tính | Giá trị |
| --- | --- |
| Method | `POST` |
| Path | `/api/auth/reset-password` |
| Auth required | Không |
| Content-Type | `application/json` |

### Request body

```json
{
  "email": "vana@example.com",
  "otp": "123456",
  "newPassword": "NewPass@123",
  "confirmPassword": "NewPass@123"
}
```

| Field | Type | Required | Rule |
| --- | --- | --- | --- |
| `email` | string | Có | Đúng định dạng email |
| `otp` | string | Có | 6 chữ số |
| `newPassword` | string | Có | ≥8 ký tự, có hoa/thường/số/ký tự đặc biệt (giống register) |
| `confirmPassword` | string | Có | Phải khớp `newPassword` |

### HTTP code

| Code | Trường hợp |
| --- | --- |
| `200` | Đổi mật khẩu thành công |
| `400` | Request không hợp lệ / OTP sai (còn lượt) / confirm không khớp |
| `404` | Không tìm thấy tài khoản ACTIVE phù hợp |
| `410` | OTP hết hạn |
| `429` | OTP nhập sai quá số lần cho phép |
| `500` | Lỗi server |

### Response thành công

```json
{
  "success": true,
  "message": "Đặt lại mật khẩu thành công. Vui lòng đăng nhập lại."
}
```

### Response OTP sai

```json
{
  "success": false,
  "message": "Mã OTP không chính xác.",
  "data": { "remainingAttempts": 2 }
}
```

## 3. Bảng / Redis liên quan

### `users`
- Đọc: tìm theo `email`, chỉ xử lý `status = ACTIVE`.
- Ghi: cập nhật `users.password = <hash mới>` khi reset thành công.

### `refresh_tokens`
- Ghi: **xóa toàn bộ** phiên của user sau khi đổi mật khẩu (logout mọi thiết bị).

### `account_security`
- Ghi: reset `failed_login_attempts = 0`, `locked_until = null` để user đăng nhập lại ngay.

### OTP — Redis (purpose mới `PASSWORD_RESET`)
- Key: `otp:PASSWORD_RESET:{email}`, Value Hash `{ code, attempts }`, TTL 300s.
- Cooldown: `otp:cooldown:PASSWORD_RESET:{email}`, TTL `OTP_RESEND_COOLDOWN_SECONDS`.
- Hoàn toàn tách khỏi key `otp:REGISTRATION:{email}` — cùng email vẫn không đụng nhau.

---

## 4. Bảo mật & quy ước

- `forgot-password` **không lộ** email có tồn tại hay không (luôn `200`).
- `reset-password`: **`DEL` key OTP trước** khi đổi mật khẩu → chống double-submit.
- Đổi mật khẩu xong **thu hồi mọi refresh token** → các phiên cũ không refresh được; access token cũ tự hết hạn sau ≤15 phút.
- OTP không bao giờ xuất hiện trong response.
- `identifier` (email) normalize giống các flow OTP khác để key trùng đúng.
- Quy tắc mật khẩu mới dùng lại đúng `passwordRegex` của register để nhất quán.

## 5. Mở rộng `resend-otp` đa purpose

Hiện `resend-otp` chỉ phục vụ `REGISTRATION` (schema ép `purpose = literal('REGISTRATION')`,
service kiểm tra user `INACTIVE`). Cần cho phép gửi lại OTP cả khi `PASSWORD_RESET`.

### Thay đổi

1. **`models/User.js` — `resendOtpSchema`:** đổi `purpose` từ `z.literal('REGISTRATION')` sang
   `z.enum(['REGISTRATION', 'PASSWORD_RESET']).default('REGISTRATION')`.
2. **`redis/otpStore.js` — `OTP_PURPOSE`:** thêm `PASSWORD_RESET: 'PASSWORD_RESET'`.
3. **`services/auth/auth.service.js` — `resendOtp({ email, purpose })`:** rẽ nhánh theo purpose:
   - `REGISTRATION`: như hiện tại — user phải `INACTIVE` (ngược lại `409`), `404` nếu không tồn tại.
   - `PASSWORD_RESET`: user phải `ACTIVE`; áp **chống enumeration** giống `forgot-password` (luôn `200`, chỉ gửi khi đúng điều kiện). Vẫn check cooldown theo `purpose`.
   - Dùng `purpose` truyền vào cho mọi lời gọi `otpStore` (set/cooldown) và `sendOtpEmail`.
4. **`controllers/auth/auth.controller.js`:** truyền `purpose` từ `req.body` xuống service (đã validate/defaulted).

> Lưu ý: cooldown và OTP key đều theo `purpose`, nên resend cho `PASSWORD_RESET` không ảnh hưởng OTP `REGISTRATION` của cùng email và ngược lại.

## 7. Điều kiện cần để thực hiện

1. Cụm đăng ký (1.1–1.3) đã chạy: `otpStore`, `mailer`, Redis, SMTP đã cấu hình.
2. Bảng `refresh_tokens` đã tồn tại (từ 1.6-logout).
3. Redis đang chạy và `REDIS_URL` đã cấu hình.

## 8. Kết quả mong đợi

1. User `ACTIVE` quên mật khẩu nhận được OTP qua email và đặt lại được mật khẩu mới.
2. Email không tồn tại / chưa active không bị lộ qua `forgot-password` (luôn `200`).
3. OTP `PASSWORD_RESET` hết hạn / sai quá số lần bị từ chối đúng mã `410`/`429`.
4. Sau khi đổi mật khẩu, mọi phiên refresh cũ bị thu hồi; đăng nhập bằng mật khẩu mới hoạt động, mật khẩu cũ không còn dùng được.
5. `resend-otp` gửi lại được OTP cho cả `REGISTRATION` lẫn `PASSWORD_RESET`, tách biệt theo purpose.

| forgot-password gọi lại ngay (<60s) | `200` (cooldown, không gửi lại) |
| reset-password OTP đúng | `200`, mật khẩu đổi, refresh token bị xóa hết |

| reset-password OTP sai quá số lần | `429` |
| reset-password confirm không khớp | `400` |
| login bằng mật khẩu mới sau reset | `200` |
| refresh bằng token cũ sau reset | `401` (đã bị thu hồi) |
