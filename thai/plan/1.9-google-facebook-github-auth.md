# 1.9. Đăng nhập mạng xã hội (Google / Facebook / GitHub)

## Mục tiêu

Cho phép user đăng nhập / đăng ký bằng tài khoản Google, Facebook hoặc GitHub (OAuth 2.0),
tái dùng hệ thống token nội bộ sẵn có (access 15m + refresh trong DB). Một user có thể liên
kết nhiều provider, và liên kết được với tài khoản local cùng email.

**Không** dùng OTP cho luồng này: provider đã xác thực email, nên user OAuth được tạo ở trạng
thái `ACTIVE` ngay.

---

## 1. Lựa chọn kiến trúc (chốt)

Có 2 cách triển khai OAuth:

| Cách | Mô tả | Đánh giá |
| --- | --- | --- |
| **A. Code-from-frontend (chọn)** | Frontend mở popup/redirect tới provider, nhận `code`, rồi **POST `code` về backend**. Backend đổi `code` → access token → profile. | Client secret nằm ở backend; cùng một pattern cho cả 3 provider; backend không phải giữ session redirect. **Khuyến nghị cho SPA.** |
| B. Redirect-based (backend) | Backend tự `GET /oauth/:provider` → redirect provider → `GET /callback` → trả token. | Phải quản lý `state`/cookie và cách trả token về SPA (redirect kèm token) rườm rà hơn. |

> Plan này triển khai **cách A**. `state`/PKCE do frontend xử lý phía client; backend chỉ nhận
> `code` + `redirectUri` đã dùng và thực hiện trao đổi token. (Cách B ghi ở [§9](#9-phương-án-thay-thế-redirect-based) nếu sau này cần.)

### Endpoint thống nhất cho cả 3 provider

| Method | Path | Auth | Body |
| --- | --- | --- | --- |
| POST | `/api/auth/oauth/:provider` | Không | `{ code, redirectUri }` |

`:provider` ∈ `google` \| `facebook` \| `github`. Response giống `login` (trả access + refresh + user).

---

## 2. Luồng hoạt động

1. Frontend lấy `code` từ provider (Authorization Code, có thể kèm PKCE) và gọi `POST /api/auth/oauth/:provider` với `code` + `redirectUri`.
2. Backend validate `:provider` hợp lệ (ngoài 3 giá trị → `400`).
3. Backend **đổi `code` lấy access token** tại token endpoint của provider (gửi kèm client_id/secret + redirectUri).
4. Backend **lấy profile** tại userinfo endpoint: `providerUserId`, `email`, `fullName`, `avatarUrl`.
   - GitHub: email có thể ẩn → gọi thêm `GET /user/emails`, lấy email `primary` + `verified`.
   - Nếu không lấy được **email đã xác thực** → `422` (yêu cầu cấp quyền email).
5. **Tìm liên kết** trong `oauth_accounts` theo `(provider, provider_user_id)`:
   - **Có** → lấy `user_id` tương ứng → sang bước 7.
6. **Chưa liên kết** → tìm user theo `email`:
   - **Có user** (local hoặc provider khác): nếu `BANNED` → `403`; ngược lại **liên kết**: insert `oauth_accounts` rồi sang bước 7. (Trust email do provider xác thực.)
   - **Không có user**: **tạo mới** trong transaction: `users` (status `ACTIVE`, role `TENANT`, password ngẫu nhiên không dùng được, `username` sinh tự động & unique, `avatar_url`) + `tenants` + `account_security` + `oauth_accounts`.
7. Kiểm tra trạng thái user: `BANNED` → `403`.
8. **Phát token** qua `issueTokens(user, { ipAddress, userAgent })` (tái dùng hàm hiện có) và ghi audit đăng nhập thành công.
9. Trả `200` với `{ tokens, user }` (giống `login`).

---

## 3. HTTP code

| Code | Trường hợp |
| --- | --- |
| `200` | Đăng nhập/đăng ký OAuth thành công |
| `400` | `provider` không hợp lệ / thiếu `code` / `code` sai-hết hạn khi đổi token |
| `403` | Tài khoản (theo email) đang `BANNED` |
| `422` | Provider không trả email đã xác thực → không thể tạo/liên kết tài khoản |
| `502` | Lỗi gọi provider (token/userinfo endpoint trả lỗi/timeout) |
| `500` | Lỗi server |

### Response thành công (giống login)

```json
{
  "success": true,
  "message": "Đăng nhập thành công.",
  "data": {
    "tokenType": "Bearer",
    "accessToken": "...",
    "accessExpiresIn": "15m",
    "refreshToken": "...",
    "refreshExpiresIn": "7d",
    "user": { "userId": "...", "fullName": "...", "email": "...", "username": "...", "role": "TENANT", "status": "ACTIVE" },
    "isNewUser": true
  }
}
```

> `isNewUser` (boolean) giúp frontend điều hướng (vd hỏi bổ sung SĐT) khi tài khoản vừa được tạo.

## 4. Cấu hình provider (endpoint + scope)

| Provider | Token endpoint | Userinfo endpoint | Scope tối thiểu | providerUserId |
| --- | --- | --- | --- | --- |
| Google | `https://oauth2.googleapis.com/token` | `https://www.googleapis.com/oauth2/v3/userinfo` | `openid email profile` | `sub` |
| Facebook | `https://graph.facebook.com/v19.0/oauth/access_token` | `https://graph.facebook.com/me?fields=id,name,email,picture` | `email,public_profile` | `id` |
| GitHub | `https://github.com/login/oauth/access_token` | `https://api.github.com/user` (+ `/user/emails`) | `read:user user:email` | `id` |

> Mỗi provider trả profile shape khác nhau → cần **adapter** chuẩn hóa về `{ providerUserId, email, emailVerified, fullName, avatarUrl }`. Google trả `email_verified`; Facebook coi như verified; GitHub lấy từ `/user/emails` (`verified: true`).

## 5. Bảng / cấu hình liên quan

### Bảng mới: `oauth_accounts`

| Cột | Kiểu | Ghi chú |
| --- | --- | --- |
| `oauth_account_id` | uuid PK | `gen_random_uuid()` |
| `user_id` | uuid FK → `users.user_id` | `onDelete('CASCADE')` |
| `provider` | enum(`GOOGLE`,`FACEBOOK`,`GITHUB`) | |
| `provider_user_id` | varchar | ID user phía provider |
| `email` | varchar nullable | email tại provider (tham khảo) |
| `created_at` | timestamptz default now | |

Ràng buộc: **unique(`provider`, `provider_user_id`)**; index `user_id`. (Tùy chọn unique(`user_id`,`provider`) nếu mỗi user chỉ 1 tài khoản/provider.)

> **Lưu ý ledger migration:** nhánh `thai` đang thiếu file `021/022` nên `npm run migrate` bị chặn (giống lúc làm `refresh_tokens`). Cách an toàn: tạo migration `024_create_oauth_accounts.js` nhưng **áp trực tiếp** vào DB như đã làm với `023` (chạy `up(db)` + chèn dòng ledger), cho tới khi đồng bộ lại migration với team.

### `users` — vấn đề `password notNullable`

Cột `users.password` đang `NOT NULL`. User OAuth không có mật khẩu. Hai hướng:

| Hướng | Cách | Đánh giá |
| --- | --- | --- |
| **Random hash (chọn)** | Tạo user OAuth với `password = hash(random 32 bytes)` — không ai đăng nhập được bằng nó. | Không phải đổi schema (tránh đụng ledger). User có thể đặt mật khẩu thật qua `forgot-password` (1.8). |

→ Dùng **random hash**. Ghi chú: login local của user OAuth sẽ luôn fail cho tới khi họ dùng `forgot-password` để đặt mật khẩu.

### `username` tự sinh (auto-generate, đổi sau)

`username notNullable + unique`. **Backend tự sinh** lúc tạo user OAuth, **không nhận username từ client trong call OAuth**:

- Sinh từ email: `base = slug(local-part email)`, trùng thì thêm hậu tố ngẫu nhiên (`base + '_' + rand4`), lặp tới khi unique.
- Đảm bảo khớp `usernameRegex` (`^[a-z][a-z0-9_.]{2,49}$`); nếu base không hợp lệ → prefix `user_`.
- User đổi username sau ở trang hồ sơ (tính năng profile, ngoài phạm vi 1.9).

> **Vì sao không cho user nhập username ngay trong call OAuth:** authorization `code` là **dùng-một-lần & hết hạn nhanh**. Nếu gộp username (thứ có thể fail vì trùng/sai regex) vào cùng request đổi code, khi username lỗi thì `code` đã bị tiêu thụ → user phải làm lại consent từ đầu. Tách auto-gen ra khỏi bước đổi code giúp login OAuth **không bao giờ fail vì username**. (Nếu sau này bắt buộc user tự chọn username ngay khi đăng ký OAuth, dùng onboarding 2 bước với `registrationToken` — xem [§9](#9-phương-án-thay-thế-redirect-based) cho hướng mở rộng.)

### `account_security`, `tenants`
Tạo kèm như `createTenantUser` hiện có (transaction).

### Env mới (`backend/.env` + `config/env.js`)
```
GOOGLE_CLIENT_ID=...        GOOGLE_CLIENT_SECRET=...
FACEBOOK_CLIENT_ID=...      FACEBOOK_CLIENT_SECRET=...
GITHUB_CLIENT_ID=...        GITHUB_CLIENT_SECRET=...
```
→ `env.oauth.google/facebook/github = { clientId, clientSecret }`.

## 6. Bảo mật & quy ước

- **Client secret chỉ ở backend**; không bao giờ trả ra client.
- Chỉ liên kết theo email khi **email đã verified** từ provider → tránh chiếm tài khoản bằng email chưa xác thực (đặc biệt Facebook/GitHub).
- User OAuth tạo ở `status = ACTIVE` (provider đã xác thực) — bỏ qua OTP.
- Tái dùng `issueTokens` + `writeLoginAudit` để OAuth login có cùng phiên/refresh/nhật ký như login thường.
- `provider` whitelist cứng (`google|facebook|github`); không nhận giá trị khác.
- Lỗi gọi provider tách mã `502` (phân biệt với lỗi nội bộ `500`).
- Gọi provider có **timeout** + bắt lỗi rõ ràng.

## 8. Điều kiện cần để thực hiện

1. Đã đăng ký OAuth app ở Google Cloud / Facebook Developers / GitHub Developer Settings, có client id/secret + redirect URI khớp frontend.
2. Bảng `oauth_accounts` đã tạo.
3. Bảng `refresh_tokens` (1.6) đã có để `issueTokens` chạy.
4. Frontend cài đặt lấy `code` (Authorization Code, khuyến nghị PKCE) và biết `redirectUri`.

## 9. Phương án thay thế (redirect-based)

Nếu không dùng frontend-code:
- `GET /api/auth/oauth/:provider` → set cookie `state` ngẫu nhiên → 302 tới consent URL provider.
- `GET /api/auth/oauth/:provider/callback?code&state` → verify `state` cookie → đổi code → cùng logic §2 từ bước 4 → **302 về frontend** kèm token (vd fragment `#accessToken=...`) hoặc set HttpOnly cookie refresh.
- Cần cấu hình `state` chống CSRF + chống open-redirect. Phức tạp hơn cách A nên chỉ dùng nếu frontend không tự xử lý code.

## 10. Kết quả mong đợi

1. Đăng nhập lần đầu bằng mỗi provider tạo user `ACTIVE` (role TENANT) + bản ghi `oauth_accounts`, trả token như login.
2. Đăng nhập lần sau bằng cùng provider nhận đúng user cũ (không tạo trùng).
3. Provider khác / login local cùng email được **liên kết** vào cùng một user (khi email verified).
4. User OAuth có thể đặt mật khẩu qua `forgot-password` rồi login local.
5. Email chưa verified / không có email → bị từ chối `422`, không tạo tài khoản rác.
6. Tài khoản `BANNED` không đăng nhập được qua OAuth (`403`).

## 11. Ghi chú test (Postman / e2e)

> OAuth khó test thuần Postman vì cần `code` thật từ provider. Gợi ý: lấy `code` qua trình duyệt (consent thủ công) rồi dán vào Postman; hoặc mock `oauthProviders` (stub `exchangeCode`/`fetchProfile`) để test logic find/link/create không phụ thuộc mạng.

| Case | Kỳ vọng |
| --- | --- |
| Provider không hợp lệ | `400` |
| code sai/hết hạn | `400` |
| Login Google lần đầu (email mới) | `200`, user mới `ACTIVE`, `isNewUser=true`, có `oauth_accounts` |
| Login Google lần 2 | `200`, đúng user cũ, `isNewUser=false` |
| Login GitHub cùng email với user Google | `200`, **liên kết** vào cùng user (2 dòng `oauth_accounts`) |
| Provider không trả email verified | `422` |
| Email thuộc user BANNED | `403` |
| Sau OAuth, dùng forgot-password đặt mật khẩu → login local | `200` |
