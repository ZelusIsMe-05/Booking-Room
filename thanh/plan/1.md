## 6. Update - Admin user management da implement

Them vao scope Thanh:

- `GET /api/admin/users`
- `GET /api/admin/users/:id`
- `PATCH /api/admin/users/:id/lock`
- `PATCH /api/admin/users/:id/unlock`
- `PATCH /api/admin/users/:id/role`
- `POST /api/admin/users/:id/password-reset`

Files:

- `backend/routes/admin/userRoutes.js`
- `backend/controllers/admin/userController.js`
- `backend/services/admin/userService.js`
- `backend/app.js` mount `/api/admin/users`

Behavior:

- List users supports `page`, `limit`, `role`, `status`, `keyword`.
- Detail includes role, profile fields, landlord/tenant existence, and account security summary.
- Lock sets status `BANNED`.
- Unlock sets status `ACTIVE` and clears login lock counters.
- Role update validates against `roles`.
- Password reset hashes `temporaryPassword`.
- Admin actions write `system_logs`.

Verification:

- Admin list/detail users pass.
- Lock missing user returns `404`.
- Invalid role returns `400`.
- Lock/unlock existing tenant pass.
- Role update to `TENANT` pass.
- Password reset pass.
- Tenant token calling `/api/admin/users` returns `403`.

Assumption:

- Assigning `LANDLORD` only updates `users.role_id`; it does not create a `landlords` row because that table requires CCCD URLs and this API contract only accepts `role`.
